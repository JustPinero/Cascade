# Performance Optimization Audit — 2026-06-09

Branch: `phase-10/10.5-migration-repair` · Stack: Next.js 16.2.2 (Turbopack) + Prisma 7 + SQLite
Build: clean. `.next/static` total ≈ **1.24 MB**. Turbopack does not emit a per-route size table, so chunk-level numbers are not available — findings are based on source review.

## Quick Wins (do these first)

### 1. Missing index on `Project.lastActivityAt` (hot ORDER BY)
**File:** `prisma/schema.prisma:38` · **Effort:** XS · **Impact:** M
Dashboard load (`GET /api/projects`), briefing (`GET /api/briefing`), and `lib/overseer-tools-query-projects.ts` all `orderBy: { lastActivityAt: "desc" }`. With no index, SQLite scans + sorts the full table on every dashboard hit. Add `@@index([lastActivityAt])` to `model Project`. Same case for `@@index([status, lastActivityAt])` since dashboard frequently filters by status.

### 2. Missing index on `ActivityEvent` for the polling feed
**File:** `prisma/schema.prisma:97-106` · **Effort:** XS · **Impact:** M
`ActivityFeed` polls `/api/activity` every few seconds (`app/components/activity-feed.tsx:125`) — that endpoint runs `findMany({ orderBy: createdAt desc, take: limit })` and the briefing route runs `where: { createdAt: { gte } }`. No index on `createdAt` or `projectId`. Add `@@index([createdAt])` and `@@index([projectId, createdAt])`. Highest-frequency query in the app.

### 3. Rate-limiter store is an unbounded `Map`
**File:** `lib/rate-limiter.ts:8` · **Effort:** XS · **Impact:** M (memory leak)
`const store = new Map<string, RateLimitEntry>()` — entries are added on every request but only overwritten on the same key after expiry. New keys (different prefixes per route) accumulate forever in a long-running dev server. Add a sweeper (`if (Math.random() < 0.01) for (...) if (now > resetAt) store.delete(...)`) or use a fixed cleanup interval.

### 4. `/api/knowledge/search` fetches every lesson then filters in JS
**File:** `app/api/knowledge/search/route.ts:23` · **Effort:** S · **Impact:** M
`findMany()` with no `where`, no `take`, then maps every row to lowercase and scores in memory. Push the filter into SQL: `where: { OR: [{ title: { contains: q } }, { content: { contains: q } }, { tags: { contains: q } }] }` and `take: 50`. Add `@@index([discoveredAt])` on `KnowledgeLesson` since list page sorts by it.

### 5. `/api/knowledge` (full list) is unbounded
**File:** `app/api/knowledge/route.ts:6` · **Effort:** XS · **Impact:** S
No `take`. Knowledge grows monotonically as harvest runs accumulate. Cap at e.g. 500 with pagination, and add `Cache-Control: private, max-age=30` since lessons rarely change between dashboard ticks.

## Findings

### Client-component bloat (architecture)
**Files:** `app/page.tsx:1`, `app/projects/[slug]/page.tsx:1`, `app/tasks/page.tsx:1`, `app/reports/page.tsx:1`, `app/playbook/page.tsx:1`, `app/knowledge/page.tsx:1` · **Effort:** L · **Impact:** M
Every top-level page is `"use client"` and does its initial fetch via `useEffect` → `fetch("/api/...")`. This means the user pays: HTML render → JS hydrate → fetch round-trip → re-render. Splitting each into a server parent (does Prisma queries directly, passes data as props) + small client child for interactive bits would cut TTFB and eliminate the per-page useEffect waterfall. `app/projects/[slug]/page.tsx` is the worst — it fires 4 sequential fetches in useEffect chains.

### `overseer-chat.tsx` is 1093 lines and ships on the dashboard
**File:** `app/components/overseer-chat.tsx` · **Effort:** M · **Impact:** M
Imports speech-recognition wrappers, sound libs, silence detector. Used on `/delamain` only (per nav). If it's ever imported from `/`, wrap in `next/dynamic(() => ..., { ssr: false })` to keep it off the initial dashboard chunk. No `next/dynamic` is used anywhere in the codebase — opportunity for `MorningBriefing`, `OverseerChat`, `CommandPanel`, and `DispatchResults` (all heavy, all conditional).

### N+1 in `/api/briefing`
**File:** `app/api/briefing/route.ts:46-61` · **Effort:** S · **Impact:** S (low-frequency endpoint)
Per-project `await getSessionLogs(p.path, 1)` in serial loop. Wrap in `Promise.all(projects.map(...))` to parallelize fs reads. Same pattern in `app/api/playbook/suggestions/route.ts:21`.

### `dispatchAll` serializes readiness checks
**File:** `lib/claude-dispatcher.ts:670-688` · **Effort:** S · **Impact:** S
`for` loop awaits `checkDispatchReadiness(project.path)` (multiple `fs.access` calls each) one project at a time. `Promise.all` would slash latency on multi-project dispatch.

### Over-fetching: `project.findMany()` returns every field
**Files:** `app/api/projects/route.ts:10`, `app/api/briefing/route.ts:31`, `app/api/hooks/validate/route.ts:22` · **Effort:** S · **Impact:** S
`projectContext` and `completionCriteria` are large markdown blobs (potentially KB each) and are not used by the dashboard or briefing. Add `select` to projection-only the fields the consumer needs.

### `ChatMessage` query has no compound index
**File:** `prisma/schema.prisma:187-188` · **Effort:** XS · **Impact:** S
`GET /api/overseer/history` filters `where: { sessionDate }` and `orderBy: createdAt asc`. There's `@@index([sessionDate])` but a compound `@@index([sessionDate, createdAt])` would let SQLite skip the sort.

### `HumanTask` query has no index on `status`
**File:** `prisma/schema.prisma:191` · **Effort:** XS · **Impact:** S
`/api/tasks` and `/api/briefing` both filter `where: { status: "pending" }`. Add `@@index([status])` or `@@index([status, priority, createdAt])` matching the orderBy in `app/api/tasks/route.ts:35`.

### Caching: no read-only API uses `unstable_cache` or `Cache-Control`
**Effort:** S · **Impact:** S
Only `OrphanBannerServer` uses `unstable_cache` (confirmed). Endpoints like `/api/playbook` (file-backed, rarely changes), `/api/templates`, `/api/knowledge/gaps` would benefit from `unstable_cache` with 60s revalidation. `/api/activity` should add `Cache-Control: private, max-age=2` so the activity-feed poller cannot hammer the DB on rapid focus events.

## Non-issues (verified)
- Prisma singleton: `lib/db.ts:11-18` correctly guards with `globalForPrisma`. Good.
- `jspdf` (~120 KB) is only imported from `lib/report-pdf.ts` → `app/api/reports/generate/route.ts`. Server-side only, never reaches the client bundle.
- `setInterval`/`addEventListener` use cases all have cleanup in their effect return — no leak.
- `Project.lastActivityAt` orderBy uses are 5 sites, all server-side; client work is cheap.
