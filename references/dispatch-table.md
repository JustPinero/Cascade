# Dispatch Table â€” first-class lifecycle row at enqueue

Today, `DispatchOutcome` is the only first-class persistence for a Cascade-launched session. The webhook reconstructs which dispatch a Stop-hook ping belongs to by querying for the most recent `ActivityEvent` with `eventType: "session-launched"` and `projectId` matching the path-derived slug (`app/api/webhook/session-complete/route.ts:106-112`). That introduces three latent bugs:

1. **Race:** webhook can arrive before the activity event commits â†’ outcome silently dropped.
2. **Duplicates:** Stop hook firing twice (network retry, buggy hook config) creates two `DispatchOutcome` rows and duplicate `humanTask` rows.
3. **Hung sessions:** queue slot is held forever because nothing knows the session was launched at time T with an expected completion deadline.

The `Dispatch` table fixes all three by being written **at enqueue time** with an idempotency key.

## Schema

```prisma
model Dispatch {
  id              String          @id @default(cuid())
  idempotencyKey  String          @unique           // primary correlation handle
  projectId       Int
  projectSlug     String
  mode            String                            // continue | audit | investigate | custom
  customPrompt    String?
  status          String          @default("queued") // queued | started | completed | failed | timeout
  prompt          String?                           // resolved prompt sent to Claude Code (post-template)
  healthAtDispatch String?                          // captured at enqueue
  expectedBy      DateTime?                         // enqueueTime + watchdog ttl
  enqueuedAt      DateTime        @default(now())
  startedAt       DateTime?
  completedAt     DateTime?
  errorMessage    String?

  outcome         DispatchOutcome?                  // 0..1, written by webhook on completion

  project         Project         @relation(fields: [projectId], references: [id])

  @@index([projectId, status])
  @@index([expectedBy])
  @@index([status])
}

// DispatchOutcome gets dispatchId; outcome is now keyed by dispatch, not by stale activity-event lookup
model DispatchOutcome {
  // ...existing fields unchanged...
  dispatchId      String?         @unique           // nullable for backfill of pre-Phase-23 rows
  dispatch        Dispatch?       @relation(fields: [dispatchId], references: [id])
}
```

`idempotencyKey` is a UUID generated by the dispatcher at enqueue, surfaced in the prompt template as `CASCADE_DISPATCH_ID`. The Stop hook reads that env from the spawned session's environment and POSTs it back. This is what makes the webhook lookup deterministic.

## Lifecycle

```
queued â†’ started â†’ completed
              â†“
            failed (spawn threw, validation refused)
              â†“
            timeout (expectedBy passed, no completion webhook arrived)
```

State transitions:

- `queued`: written by the dispatch route handler before queue.enqueue.
- `started`: written by the dispatcher right before `child_process.spawn`. Sets `startedAt`.
- `completed`: written by the webhook on receipt of a Stop-hook ping. Sets `completedAt` and creates the linked `DispatchOutcome`.
- `failed`: written if the dispatcher catches an error from `spawn` or pre-flight validation. Sets `completedAt` and `errorMessage`.
- `timeout`: written by a watchdog scan that runs periodically (e.g. every 5 minutes) and flips `queued`/`started` rows past `expectedBy` to `timeout`. Releases the queue slot via `getDispatchQueue().release(idempotencyKey)`.

## Idempotency rules

- The webhook upserts by `dispatchId` (or `idempotencyKey`). Duplicate Stop-hook arrivals find the existing `Dispatch` and **no-op** if status is already `completed`. They never produce a second `DispatchOutcome` or a duplicate `humanTask`.
- Human tasks created from `[HUMAN TODO]` signals are deduplicated by `(dispatchId, signal.message)` â€” same signal arriving twice, same task.

## Webhook update path (after migration)

```
1. Read body.idempotencyKey (from the Stop hook env passthrough)
   - Fallback to body.projectPath for legacy hooks during transition
2. Find Dispatch by idempotencyKey
   - If not found AND projectPath fallback resolves a recent queued/started Dispatch â†’ use it
   - If still not found â†’ log "orphaned webhook" and return 200 (don't 500 â€” we still ran the project import)
3. If status === "completed" â†’ return 200 with { ok: true, deduped: true }
4. Otherwise: update status â†’ completed, set completedAt, create DispatchOutcome with dispatchId set
5. Run escalation detection, create human tasks (deduped on (dispatchId, message))
6. Release queue slot via getDispatchQueue().release(dispatch.idempotencyKey)
```

## Migration ordering

The Dispatch table lands first. The webhook keeps its current "find latest session-launched event" path **as a fallback** through one transition window â€” old in-flight dispatches launched pre-migration won't have a `Dispatch` row, but they should still complete cleanly. After one full release cycle the fallback is removed and the webhook becomes idempotency-key-only.

## Hooks side

The Cascade Stop-hook template (under `templates/`) needs to read `$CASCADE_DISPATCH_ID` from the environment and include it in the POST body. This is a one-line change to the hook script. Existing managed projects need to re-run the hook installer, which `create-cascade` already handles for fresh installs and which a one-off `pnpm tsx scripts/refresh-hooks.ts` script can sweep for existing fleets.

## What this enables

- **Dashboard "in-flight" tile**: count of Dispatch rows with status in (`queued`, `started`). Today the only signal is the queue's RAM count, which doesn't survive process restart.
- **Outcome-conditioned dispatch (Phase 24)**: the Overseer can read `Dispatch` rows to see "I dispatched audit on medipal twice in the last week, both produced no signals" â€” that's exactly the dataset for proposing `continue` instead.
- **Hung-session telemetry**: a graph of `timeout` rate per project surfaces brittle hooks or stuck Claude Code sessions immediately.
