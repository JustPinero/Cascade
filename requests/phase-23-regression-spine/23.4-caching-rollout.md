# Request 23.4 — Prompt Caching Rollout

## Objective

Wire `cache_control` markers into every Cascade Anthropic call site where the prefix is stable enough to benefit, then snapshot-test those prefixes so silent drift fails CI loudly. The single highest-leverage marker is on the **last tool** returned by `ToolRegistry.toAnthropicTools()` — Sonnet 4.6's cache minimum is 2,048 tokens and the Overseer's system prompt alone (~1,700 tokens) is below it, so caching the system+tools combined block is the only way the Overseer prefix exceeds the threshold.

Other rollouts: cache the synthetic compressed-history summary block, cache the per-project chat's dynamic-but-session-stable system prompt, and cache the feature proposer's static system prompt with a 1-hour TTL. Skip the Haiku summarizer (its system prompt is ~100 tokens, well below Haiku 4.5's 4,096 minimum) and the briefing endpoint (entirely dynamic).

This slice is small in code and large in care — every change is one or two lines, but each one is load-bearing for cost and latency. The snapshot tests are what keep them load-bearing in the future.

## GREEN Phase (implementation)

Files to create:

- `lib/overseer-tools.types.test.ts` — pure type-level tests using `expectTypeOf` (vitest) or assignment-to-typed-variable patterns. Tests 1-4.
- `lib/__tests__/prompt-snapshots.test.ts` — the canonical snapshot file. Tests 10-14. Uses `toMatchSnapshot()`. Snapshot files committed to the repo. Diffs are reviewed in the PR.
- `lib/__tests__/caching-end-to-end.test.ts` — integration test 15. Stubs fetch, asserts both the outbound request shape and the inbound usage row. Lightweight — does not use the dispatch rig.

Files to modify:

- `lib/overseer-tools.ts`:
  - Widen `TextBlock`: `{ type: "text"; text: string; cache_control?: { type: "ephemeral"; ttl?: "1h" } }`.
  - Widen `AnthropicToolDefinition`: add optional `cache_control` of the same shape.
  - Widen `AnthropicMessageParams.system` to `string | Array<{ type: "text"; text: string; cache_control?: ... }>`. Define a named type `SystemBlock` for clarity.
  - Widen `AnthropicMessageResponse.usage` to include the optional cache fields and the optional `cache_creation: { ephemeral_5m_input_tokens?, ephemeral_1h_input_tokens? }` split. (May already partially exist after 23.3; verify and complete.)
  - Inside `ToolRegistry.toAnthropicTools()`, after building the array, if `tools.length > 0`, set `tools[tools.length - 1].cache_control = { type: "ephemeral" }`. Comment notes this is the cache-prefix marker per `references/prompt-caching.md`.
- `lib/chat-history-compressor.ts`:
  - Update `formatSummaryAsMessage` to return `{ role: "user", content: [{ type: "text", text: "...", cache_control: { type: "ephemeral" } } as ContentBlock] }`.
  - The truncation-notice variant (`formatTruncationNotice`) does NOT get cache_control — the truncation message is not stable enough to warrant caching, and its content reflects an error path.
- `app/api/projects/[slug]/chat/route.ts`:
  - Build the system prompt as before. Wrap the result: `const system: SystemBlock = [{ type: "text", text: built, cache_control: { type: "ephemeral" } }]`. Pass to the Anthropic call.
  - The route already streams (`stream: true`); caching combines with streaming per `references/prompt-caching.md`. No streaming-specific change needed.
- `lib/anthropic-feature-proposer.ts`:
  - Wrap the static system prompt: `system: [{ type: "text", text: PROPOSAL_SYSTEM_PROMPT, cache_control: { type: "ephemeral", ttl: "1h" } }]`.
  - Comment notes the 1h TTL is intentional — `proposeForAll()` calls cluster within minutes, and the cost asymmetry (2.0× write but 0.1× read) pays off when ≥3 features proposed in a burst.
- `lib/anthropic-feature-check.ts` and other call sites — verify no accidental cache_control additions; this slice is targeted, not a sweep.

## Post-deploy verification

After the slice merges, run a real Overseer turn against production. Within 5 minutes, run a second turn. Open `/observability/cache`. Confirm:

- First call: `cacheCreationInputTokens > 0`, `cacheReadInputTokens == 0`. Cache write happened.
- Second call: `cacheReadInputTokens > 0`, `cacheCreationInputTokens` low or 0. Cache hit confirmed.
- Hit rate column shows ≥50% on the second row.

Same check for per-project chat after two turns in the same project session. If feature proposer doesn't show `cacheCreationInputTokens > 0` on its first call, its system prompt is below the 2,048 minimum — log this in `audits/debt.md` and either expand the prompt or remove the cache_control marker (no point shipping a no-op marker that misleads future readers).

This verification step is a manual check, not a test. Capture the result in `.claude/handoff.md` after running it.

## Dependencies

- 23.3 (Anthropic usage telemetry) — required. Without telemetry, post-deploy verification is impossible and silent regressions go undetected.
- 23.1 (Dispatch Rig) — not required. Tests in this slice use lightweight inline mocks, not the rig.

## Out of scope

- **Streaming call sites' usage logging.** Wizard chat and per-project chat both stream their Anthropic responses to the client. The caching markers in this slice are correct (caching works with streaming), but extracting `usage` from a streamed response requires a TransformStream wrapper. Deferred to land alongside Phase 25.2. Until then, those call sites' cache hits are invisible in `/observability/cache` even though the caching itself is working.
- **Skipping the cache_control marker on the feature proposer if its prompt is small.** The post-deploy check tells the truth; we ship the marker either way and adjust based on telemetry.
- **Caching on the Haiku summarizer** — system prompt ~100 tokens, well below Haiku 4.5's 4,096 minimum. No marker added; reference doc explains.
- **Caching on the briefing endpoint** — entirely dynamic content per request; no stable prefix worth marking.
- **Pre-warming the cache via `max_tokens: 0`** — not relevant for Cascade's request shapes.
- **Adjusting the Overseer's `maxIterations: 8` ceiling.** Tool-loop length is unrelated to caching; the cache prefix survives all 8 iterations because the system+tools block is identical each turn.
- **An Overseer `query_cache_stats` tool.** Out of 23.4. The `/observability/cache` page covers the use case; a tool is feature creep here.
