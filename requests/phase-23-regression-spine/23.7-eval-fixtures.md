# Request 23.7 — Initial Eval Fixtures

## Objective

Seed the eval suite with the first batch of fixtures so `pnpm eval` actually runs something. Three fixture kinds, one runner from 23.6:

- **5 Overseer scenarios** — each defines DB preconditions, input messages, and an asserted tool sequence + final-text matcher. These exercise the model's behavior on representative Cascade workflows (inventory walks, blocker triage, dispatch proposals, knowledge queries, fleet status). Each requires a captured Anthropic recording.
- **3 knowledge-matcher scenarios** — each gives an issue description plus a lesson corpus and asserts the top-3 returned IDs. No Anthropic call; the matcher is pure TypeScript.
- **30+ escalation-detector logs** — a synthetic-but-realistic corpus covering the regex's positive cases and known false-positive risks. No Anthropic call.

Once this slice lands, future regressions in the model's tool selection, the matcher's ranking, or the detector's regex coverage will turn red in CI. The whole reason to invest in the eval runner is that this corpus exists.

## RED Phase (failing tests)

The TDD shape here is unusual — fixtures are themselves the tests. Each fixture starts failing (no recording yet, or asserter mismatches the live model's output, or `expected.json` is wrong) and goes green when the fixture is finalized.

- [ ] Tests 1, 5, 7: each filesystem assertion fails because the files don't exist yet
- [ ] Test 2: write each Overseer fixture as JSON; run schema validation and watch it fail until the JSON shape is correct
- [ ] Test 3: each fixture has no recording on first authoring; run `pnpm eval:refresh:scenario=...` to capture it
- [ ] Test 4: with recordings present, replay-run each fixture and watch the asserter fail until the expected tool sequence matches what the model actually does. Tune `inputContains` matchers, allow flexibility in `minToolCalls`, until the assertion accepts real model behavior without being permissive enough to mask regressions
- [ ] Test 6: write the 3 matcher fixtures with a chosen lesson corpus; run; watch the assertion fail until expected IDs match what the matcher returns. If the matcher's behavior is wrong (lessons in wrong order, missing critical match), fix the matcher in this slice or log to debt
- [ ] Tests 7-11: write each `.log` + `expected.json` pair; run the detector against it; watch failures surface where the regex doesn't catch what we expected, or catches things we didn't expect. Fix the regex if reasonable, or pin the corpus to the regex's actual behavior with a comment explaining why
- [ ] Test 12: assert `pnpm eval` exits 0 with all 38+ scenarios reported as passing
- [ ] Test 13: capture a recording, then re-run refresh against the same scenario; assert the file at the same hash path is rewritten (or unchanged if no API drift) and no new file is created at a different hash

Path: `evals/scenarios/overseer/`

- `inventory-walk-medipal.json` — preconditions: 1 project (`medipal`, phase-5, healthy). Input: "let's do a fleet inventory walk, start with medipal". Expected tool sequence: `set_active_flow({flow: "inventory_walk"})` → `query_project({slug: "medipal"})`. `finalTextMatches: "/medipal/i"`. `minToolCalls: 2`.
- `dispatch-after-stall.json` — preconditions: 1 project with low `progressScore` and 5+ recent `session-launched` activity events but no `phase-complete`. Input: "anything stalled lately?". Expected: `query_projects` → optionally `get_recent_activity` → final text references the project name and the word "stalled" or "stuck".
- `blocker-triage.json` — preconditions: 1 project with `health: "blocked"` and a recent activity event of type `blocker-detected` whose summary contains "[NEEDS ATTENTION] migration failed". Input: "what's blocking me?". Expected: `query_projects({status: "blocked"})` or `get_recent_activity` → final text quotes "migration failed".
- `knowledge-query.json` — preconditions: 3 `KnowledgeLesson` rows, one with `category: "deployment"` titled "SQLite WAL mode false locks". Input: "what have we learned about deployment?". Expected tool sequence depends on the registered knowledge tool — for now, assert `finalTextMatches: "/SQLite|WAL/"` and `minToolCalls: 1`. (When 25.3 adds `query_knowledge_with_citations`, this fixture is updated to assert that tool specifically.)
- `fleet-status-quick.json` — preconditions: 4 projects across mixed states (1 healthy, 1 warning, 1 blocked, 1 idle). Input: "fleet status". Expected: `query_projects` → final text matches `/active|idle|blocked/` and includes a count.

For false-positive logs, `expected.json` is `{ "signals": [] }` — explicitly assert no false positive.

## Authoring workflow (worth pinning)

Each Overseer fixture goes through the following loop on first authoring:

1. Write the fixture JSON with your best guess at the expected tool sequence.
2. Run `pnpm eval:refresh:scenario=overseer/<name>` to capture the recording.
3. Run `pnpm eval --scenario=overseer/<name>` (replay) and watch the asserter compare your expectations against what the model actually did.
4. Iterate: tighten or loosen the asserter (e.g. add `inputContains`, raise `minToolCalls`, adjust `finalTextMatches`) until the asserter accepts real model behavior without becoming permissive enough to miss regressions.
5. Commit the fixture and the recording in the same PR.

The bar for "asserter is right":
- Stricter is better (catches more regressions) but flaky if the model has variance.
- Loose enough that re-recording with the same prompt produces a passing result.
- Strict enough that swapping `claude-sonnet-4-6` → `claude-haiku-4-5` would likely fail.

## Out of scope

- **Real-world session logs in the escalation corpus.** All 30 logs are synthetic. A future slice can add 5-10 sanitized real logs from the user's fleet to catch patterns the synthetic corpus misses. Track this in `audits/debt.md`.
- **Re-recording on every PR.** Recordings are point-in-time snapshots; re-recording is opt-in via `pnpm eval:refresh`. PRs that change Overseer behavior (system prompt, tools, model) are expected to refresh and review the diff.
- **Cross-fixture preconditions.** Each scenario seeds its own DB. No "shared base seed" across scenarios.
- **Eval scenarios for the project chat or wizard endpoints.** Future slice. The Overseer is the load-bearing AI surface today.
- **Eval scenarios for outcome-conditioned dispatch (Phase 24)** or knowledge-citations (Phase 25) — those phases ship their own fixtures.
- **Fuzzing the escalation regex with a generator** — the corpus is hand-curated. A property-based variant is a future option if the corpus stops catching new failures.
