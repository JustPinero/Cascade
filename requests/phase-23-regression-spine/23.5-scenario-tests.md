# Request 23.5 — Scenario Tests

## Objective

Land the catalog of system-level scenario tests that exercise the dispatch service's known failure modes. Each is a 10–30 line story written on top of the rig from 23.1, asserting that a specific failure mode is handled correctly given the dispatch-table fixes from 23.2.

This slice does not add production code. The rig (23.1) is the harness; the dispatch table (23.2) is the fix; this slice is the catalog of what we now guarantee. If a future change reintroduces a race, a leak, or a swallowed error, one of these tests should turn red.

The phase plan listed 12 scenarios; two of those (`webhook-fires-twice` and `webhook-human-task-dedup`) were absorbed into 23.2's tests because they're the critical path for the idempotency-key migration. This slice covers the remaining 10.

## RED Phase (failing tests)

Each test must fail before the rig is in place, fail before the dispatch table migration is in place, or fail loudly if a regression is introduced.

- [ ] Test 1: enqueue a dispatch via the rig; manually delete the matching `session-launched` `ActivityEvent` row before firing the webhook; fire the webhook with `{ projectPath, idempotencyKey: dispatch.idempotencyKey }`; assert the Dispatch is `completed` and one `DispatchOutcome` row exists (`tests/scenarios/webhook-completes-without-activity-event.test.ts`)
- [ ] Test 2: configure `rig.setExecSyncBehavior(/respawn-pane/, ...)` to throw on the first matching call only; trigger a `dispatchBatch` of 3 projects; assert the failing pane's queue slot is released, an error is logged, and the other 2 dispatches reach `status: "started"`. Assert no orphan running-count is leaked (`rig.queue.size().running` matches expectations) (`tests/scenarios/respawn-pane-fails.test.ts`)
- [ ] Test 3: cap the queue at 1; enqueue 2 dispatches in sequence — first with `spawnThrows: new Error("boom")`, second with default success; assert second's Dispatch row reaches `status: "started"` and `rig.spawnRecords.length === 2` (`tests/scenarios/dispatch-throws-next-runs.test.ts`)
- [ ] Test 4: enqueue a dispatch; capture the project's id; delete the Project row directly via `rig.prisma.project.delete`; fire the webhook for the original projectPath; assert the response is 200, no exception thrown, and an activity event with `eventType: "orphaned-webhook"` (or whatever the implementation chooses) exists (`tests/scenarios/webhook-after-project-deletion.test.ts`)
- [ ] Test 5: configure rig with concurrency cap 2; enqueue two dispatches for the same project via `Promise.all([rig.beginDispatch(...), rig.beginDispatch(...)])`; assert both `Dispatch` rows exist with distinct `idempotencyKey`s, both have `status: "started"`, and `rig.spawnRecords.length === 2` (`tests/scenarios/concurrent-dispatch-same-project.test.ts`)
- [ ] Test 6: enqueue a dispatch; do NOT fire the webhook; advance fake time by `31 * 60 * 1000` (just past the default 30-minute `expectedBy`); call `runDispatchWatchdog(rig.prisma, rig.queue)`; assert Dispatch.status is `timeout`, `rig.queue.size().running === 0`. Then enqueue a fresh dispatch and assert it reaches `status: "started"` immediately, proving the slot was reclaimed (`tests/scenarios/hung-session-watchdog.test.ts`)
- [ ] Test 7: stub `importSingleProject` to throw; enqueue a dispatch and fire the webhook; assert the handler returns 200, the Dispatch row's `status === "completed"`, an error is logged via console.error or activity event, and no exception escapes the handler (`tests/scenarios/import-fails-mid-webhook.test.ts`)
- [ ] Test 8: race `getOrCreateSession(rig.prisma, "2026-05-04")` 5 times in `Promise.all`; assert all returned ids match and `rig.prisma.chatSession.count({ where: { id: ... } }) === 1` (`tests/scenarios/concurrent-chat-session.test.ts`)
- [ ] Test 9: dispatch with `customPrompt: "Test \"with quotes\" and $vars and \`backticks\` and \\backslashes and\nnewlines."`; assert `rig.spawnRecords[0]` reflects an unmangled command (no broken JSON, no shell errors) and `rig.prisma.dispatch.findFirst({ where: { idempotencyKey } }).prompt` round-trips the input (`tests/scenarios/osascript-quote-injection.test.ts`)
- [ ] Test 10: enqueue a dispatch for project "alpha" at path `/tmp/alpha`; rename the project (update slug to "alpha-renamed", path to `/tmp/alpha-renamed`); fire the webhook with the original `/tmp/alpha`; assert response 200 and an `orphaned-webhook` event is logged (`tests/scenarios/project-renamed-mid-session.test.ts`)
- [ ] Test 11: stub `rig.prisma.dispatchOutcome.create` to throw; enqueue + fire webhook; assert response 200, Dispatch row's `status === "completed"`, error logged, no `DispatchOutcome` row exists (`tests/scenarios/dispatch-outcome-create-fails.test.ts`)

In short: the production code fixes for tests 1, 2, 3, 5, 6, 8, 10 should already be in place after 23.1 + 23.2. Tests 4, 7, 9, 11 may surface small refactors that this slice ships alongside the test.

Each refactor is no more than ~10 lines of code. If a scenario test surfaces a larger fix, log it in `audits/debt.md` and ship a minimal patch here that satisfies the test, with the fuller fix tracked separately.

## Out of scope

- **`webhook-fires-twice` and `webhook-human-task-dedup`** — already covered in 23.2 tests 9 and 10.
- **Performance / load tests** — these are correctness scenarios, not throughput. Concurrency tests assert "no double-creates," not "scales to N concurrent dispatches per second."
- **End-to-end UI tests for the same scenarios** — Playwright covers UX flow at the browser level. These scenario tests are server-side only.
- **Fuzzing or property-based tests** — every scenario is hand-written for a known failure mode. If a scenario class repeatedly catches new bugs (e.g. shell-escape handles `'` but not `;`), revisit with a property-based generator in a follow-up slice.
- **Replacing the rig's mocks with real `child_process` / `osascript` calls** — scenario tests stay hermetic. Real-world dispatch behavior is covered by E2E in 23.8.
- **Scenario tests for the Anthropic API side** — those live in evals (23.7), not scenarios.

## Definition of done

- All 11 test files exist under `tests/scenarios/` and pass.
- Each test fails when its corresponding production fix is reverted (verify by manually un-applying each fix and re-running).
- `pnpm test tests/scenarios/` runs in under 30 seconds total — fake timers and in-memory SQLite keep them fast.
- No new flake. If any test relies on real timing rather than `vi.useFakeTimers()`, fix before merge.
