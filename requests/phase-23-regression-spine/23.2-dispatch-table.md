# Request 23.2 — Dispatch Table + Idempotency Migration

## Objective

Introduce a first-class `Dispatch` row written at enqueue time with an idempotency key, plumb that key through the spawned session as `CASCADE_DISPATCH_ID`, rewire the webhook to update by key (with a legacy fallback for orphaned hooks during the transition window), and add a time-based watchdog that flips abandoned dispatches to `timeout` and releases their queue slots.

This single slice fixes three latent bugs at once: (a) the webhook race where a Stop hook arrives before the activity event commits and the outcome is silently dropped; (b) duplicate webhooks creating duplicate `DispatchOutcome` and `humanTask` rows; (c) hung sessions holding queue slots forever. After this lands, every code path between enqueue and outcome is keyed to a stable identifier instead of a stale-event lookup.

Schema, lifecycle, and migration ordering are specified in `references/dispatch-table.md`. Tests use the rig from 23.1.

## GREEN Phase (implementation)

Files to create:

- `lib/dispatch-watchdog.ts` — exports `runDispatchWatchdog(prisma, queue, opts?)`. Default scan window: rows with `status in ("queued","started")` and `expectedBy < now`. Behavior per row: update to `status: "timeout"`, set `completedAt`, fire `getDispatchQueue().release(idempotencyKey)`, write an `ActivityEvent` of type `dispatch-timeout`. Idempotent: rows already at `timeout` are skipped.
- `lib/dispatch-watchdog.test.ts` — tests 13-16.
- `tests/scenarios/dispatch-table-write-at-enqueue.test.ts` — test 4.
- `tests/scenarios/dispatch-table-transitions.test.ts` — test 5.
- `tests/scenarios/dispatch-table-spawn-fails.test.ts` — test 6.
- `tests/scenarios/dispatch-table-env-passthrough.test.ts` — test 7.
- `tests/scenarios/webhook-idempotency-key-path.test.ts` — test 8.
- `tests/scenarios/webhook-fires-twice.test.ts` — tests 9 and 10 (or split if cleaner).
- `tests/scenarios/webhook-human-task-dedup.test.ts` — if not folded into 9.

- `templates/__tests__/stop-hook.test.ts` — test 17. Uses the templates' actual file content; if rendering requires a small helper (e.g. variable substitution), put it in a `templates/lib/render.ts`.
- `scripts/refresh-hooks.ts` — sweeps `PROJECTS_DIR` (or `~/projects` default) and rewrites `.claude/settings.json` Stop hook entries to the new shape. Idempotent — re-running on already-updated projects is a no-op.
- `scripts/refresh-hooks.test.ts` — test 18.

Files to modify:

- `prisma/schema.prisma` — add the `Dispatch` model and `dispatchId` field on `DispatchOutcome` per `references/dispatch-table.md`. Run `pnpm exec prisma db push` and commit the regenerated client output.
- `lib/claude-dispatcher.ts`:
  - Generate `idempotencyKey: cuid()` at the top of `dispatchClaude` (and equivalent entry points for `dispatchBatch`, `dispatchTeam`).
  - Insert the `Dispatch` row with `status: "queued"`, `enqueuedAt: now`, `expectedBy: now + opts.watchdogTtlMs ?? 30 * 60 * 1000` before `queue.enqueue`.
  - Inside the queued job: just before `spawn`, update the row to `status: "started"`, `startedAt: now`. On a thrown spawn, update to `status: "failed"` with `errorMessage` and `completedAt`, then re-throw so the queue's existing error path runs.
  - Pass `env: { ...process.env, CASCADE_DISPATCH_ID: idempotencyKey }` in the spawn options. Same for the `osascript` and `tmux respawn-pane` paths — they need the env injection.
- `app/api/webhook/session-complete/route.ts`:
  - Parse `idempotencyKey` from body. If present and matches a `Dispatch` row: branch into the new path (find by key, dedup if already completed, update status, create outcome with `dispatchId`).
  - If absent or unknown: log a `dispatch-orphaned-webhook` activity event and run the existing legacy lookup. The legacy path stays for one transition window.
  - Wherever `humanTask.create` runs, key the dedup on `(dispatchId, signal.message)` — fold into a `findFirst` + skip pattern.
  - On dedup-no-op return, response shape is `{ ok: true, deduped: true, slug, name }`.
- `app/api/dispatch/all/route.ts` and `app/api/projects/[slug]/dispatch/route.ts` — no direct change required if `dispatchClaude` now writes the row internally; verify via test 4.
- The Stop hook template under `templates/` (the file that gets installed into managed projects' `.claude/settings.json`) — extend it to read `$CASCADE_DISPATCH_ID` and include `idempotencyKey` in the POST body when set. Old behavior preserved when env is unset (the legacy fallback covers it).
- `tests/harness/dispatch-rig.ts` (from 23.1) — `fireWebhook({ projectPath, sessionId?, idempotencyKey?, logContent? })` already exists per the rig's reference; verify the `idempotencyKey` field is wired into the constructed body.

## Out of scope

- Removing the legacy "find latest session-launched activity event" fallback path. That removal is a future cleanup once production telemetry confirms zero orphaned-webhook events for a sustained window.
- Wiring `runDispatchWatchdog` into a scheduled Next.js cron / interval. The function is callable; tests invoke it directly. Scheduling is a separate small slice (or rolls into 23.9 with `scripts/validate.sh`).
- A dashboard tile showing in-flight Dispatch rows. Useful follow-up but not required for the bug fixes this slice ships.
- Auto-applying outcome history to dispatch decisions — Phase 24.
- Backfilling `dispatchId` on historical `DispatchOutcome` rows. They stay null; analytics that need outcome-by-dispatch correlation simply ignore pre-23.2 rows.
