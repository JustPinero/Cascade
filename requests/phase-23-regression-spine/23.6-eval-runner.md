# Request 23.6 — Eval Runner Infrastructure

## Objective

Build the behavioral regression infrastructure for Cascade's AI surface: a recorded-response runner, a request-body hasher, three asserters, and a CLI surface. The actual eval fixtures (Overseer scenarios, knowledge-matcher fixtures, escalation corpus) ship in 23.7 — this slice is scaffolding only.

The runner replays recorded Anthropic responses by default so PR CI is deterministic, free, and offline. `pnpm eval:refresh` flips it to live-API mode for re-recording when prompts or models change. Recordings are committed alongside fixtures so a refresh diff is reviewable evidence of behavior change.

Architecture and fixture shape are specified in `references/eval-runner.md`.

## Hash function specification (worth pinning)

- Input: the Anthropic API request body as a JS object.
- Step 1: deep-clone (don't mutate caller's object).
- Step 2: walk the tree; for any block with a `cache_control` key, delete the key. Specifically: every entry in `tools` may have it; every text block in `system` may have it; every text block inside any message's `content` array may have it.
- Step 3: stringify with sorted keys (recursive). Use a deterministic JSON serializer — e.g. write a small helper that walks objects and arrays, sorting object keys alphabetically before serializing.
- Step 4: SHA-256 the resulting string; hex-encode; take the first 16 chars.

The 16-char prefix is enough collision-resistance for an eval suite (2^64 possible values, ~hundreds of recordings). Keeps filenames readable.

## Dependencies

- None. This slice is pure infrastructure; the existing call-site interfaces (`AnthropicCaller`, etc.) already accept injected callers, so no production-code changes are required.
- 23.7 (Initial eval fixtures) **depends on this slice.**

## Out of scope

- **The eval fixtures themselves** — Phase 23.7. Without fixtures, `pnpm eval` runs zero scenarios and exits 0.
- **Cost tracking on `pnpm eval:refresh`** — refreshing eval recordings hits the live API and bills it. A future slice can add per-refresh cost reporting using the telemetry from 23.3, but it's not required to ship the runner.
- **Eval scenario authoring tools** — no GUI, no template generator. Scenarios are JSON written by hand or by copying an existing one.
- **Cross-scenario sharing of preconditions** — each scenario seeds its own DB. If many scenarios share a base seed, the duplication is intentional for now (each scenario stays self-contained).
- **Parallelization** — the runner runs scenarios serially. Adding concurrency invites flake from shared module state (e.g. `getDispatchQueue` singleton). Defer until proven necessary.
- **Comparing recorded responses to live ones for drift detection** — the workflow is "refresh and review the diff in PR." A separate "drift report" command is a future enhancement.
- **Recording responses for non-Anthropic APIs.** Cascade only calls Anthropic; if that ever changes, the recorder generalizes via the same hash-then-store pattern.
