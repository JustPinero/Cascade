# Request 23.1 — Dispatch Rig

## Objective

Build the shared test harness at `tests/harness/dispatch-rig.ts` so every subsequent Phase 23 scenario test reads as a 10-line story instead of re-mocking `child_process`, `execSync`, fs, fetch, and Prisma in every file. The rig owns the boring plumbing: scratch SQLite per rig, spawn/execSync mocks with controllable behavior, direct invocation of the webhook route handler, fake timers with `advanceTime(ms)`, full cleanup via `dispose()`. Public surface, mock policy, and fixture layout are specified in `references/dispatch-rig.md`.

The rig is testing infrastructure, so TDD here has a meta quality: write tests that assert the rig itself does what it claims (isolated DB, mocked spawn records calls, fake timers advance, dispose restores state), then implement until those tests pass. As a final sanity check that the rig is real and not theater, port one existing test (`lib/claude-dispatcher.multi.test.ts`'s "can dispatch multiple modes" case) onto the rig and confirm it still asserts the same behavior.

## GREEN Phase (implementation)

Implement until every test above passes. Files to create:

- `tests/harness/dispatch-rig.types.ts` — public types: `DispatchRig`, `BeginDispatchOpts`, `PendingDispatch`, `SpawnRecord`. Mirror the reference doc's surface exactly.
- `tests/harness/dispatch-rig.ts` — `createDispatchRig(opts?)` plus a small `pushTestSchema(prisma)` helper if not already exposed by `lib/db`. Internals:
  - Scratch SQLite via `file:./test-dispatch-${randomId}.db?mode=memory&cache=shared` (fall back to a real file if the in-memory shared-cache path doesn't survive multiple Prisma clients in the same process — verify in test 2 that two rigs are isolated).
  - Reset `getDispatchQueue` via `__resetDispatchQueueForTests()` and construct a `DispatchQueue` at the requested concurrency (default 1).
  - Install `vi.useFakeTimers()` unless `opts?.fakeTimers === false`.
  - Mock `child_process.spawn` and `child_process.execSync` via `vi.spyOn` or module mock. Track calls in `rig.spawnRecords` and `rig.execRecords`.
  - Implement `mockAnthropicResponse(handler)` that registers a fetch interceptor. If no handler is registered when fetch hits `api.anthropic.com`, throw "Anthropic API called without mock — this test is missing rig.mockAnthropicResponse(...)".
  - Implement `fireWebhook` by importing the route's `POST` handler and constructing a NextRequest stub. The handler runs against `rig.prisma` — wire this via the existing `prisma` import path (likely needs `vi.mock("@/lib/db", () => ({ prisma: rig.prisma }))` set up at rig-construction time).
  - `dispose()` runs cleanup in reverse order. Idempotent — calling it twice does not throw.
- `tests/harness/fixtures/project-skeletons/cascade-test-project/` — minimal directory tree that passes `importSingleProject`: `CLAUDE.md` (one line), `.git/` (empty `HEAD` and `config` files), `.claude/handoff.md` (one line). Used by `rig.createProject({ slug, path? })` when `path` is omitted.
- `lib/claude-dispatcher.multi.test.ts` — **rewritten in place**. Same file path, same test names, same observable assertions. The implementation underneath swaps from hand-rolled mocks to `createDispatchRig`. Old mock scaffolding is removed in the same commit; nothing parallel-runs.

Files to modify:

- `lib/dispatch-queue.ts` — already exports `__resetDispatchQueueForTests`. No change expected; verify.
- `lib/db.ts` — if `pushTestSchema` is not exposed, add a thin export `pushTestSchema(prisma)` that runs the SQL needed to create all tables. (If the existing test files already do this inline, lift the helper here so the rig doesn't duplicate.)
