# Phase 23 — Regression Spine + Caching

## Why

Cascade has 297 unit tests, an excellent test:source ratio in `lib/`, and zero open debt. What it does not have:

- **System-level coverage of the dispatch service.** Each piece is unit-tested; the path from enqueue → spawn → Stop hook → DispatchOutcome write is not. Almost every dispatch bug we feel in production lives in that span. The webhook reconstructs which dispatch a Stop-hook ping belongs to via a stale-event lookup that races, duplicates, and silently drops outcomes.
- **A behavioral safety net for the AI parts.** Nothing currently tells us that the Overseer regressed when a system prompt edits, a tool description changes, or the model ID flips. 297 unit tests of `false` would also be 297 green tests.
- **Prompt caching.** The Overseer ships a stable ~7-10K-token prefix (system + 14 tools) on every turn and pays full price every time. Sonnet 4.6's cache minimum is 2,048 tokens — the system prompt alone is below that, but system+tools combined is well past it. The `cache_control` marker is one line; the win is real cost and latency reduction.
- **Observability of the AI surface.** No telemetry on cache hit rate, no telemetry on tool selection, no `/observability` page. Tuning blind.

Phase 23 is the rigorous one. It builds the regression spine (rig, scenario tests, eval runner, snapshots), wires telemetry, lands prompt caching, and re-enables E2E in CI. After this phase, prompt drift, model regressions, and dispatch races all have tests that catch them.

### 23.2 — Dispatch table + idempotency migration

- Schema change per `references/dispatch-table.md`: new `Dispatch` model, add `dispatchId` to `DispatchOutcome`, indices on `[projectId, status]`, `[expectedBy]`, `[status]`.
- `prisma db push` to sync; commit the regenerated client.
- Update `lib/claude-dispatcher.ts` to write a `Dispatch` row (status `queued`) at enqueue and flip to `started` right before `spawn`. Generate `idempotencyKey` (cuid). Plumb it into the prompt environment as `CASCADE_DISPATCH_ID`.
- Update the Stop-hook template under `templates/` to read `$CASCADE_DISPATCH_ID` and include it in the webhook POST body.
- Update `app/api/webhook/session-complete/route.ts` to find the `Dispatch` row by `idempotencyKey` first; fall back to the legacy session-launched lookup. On duplicate webhook (status already `completed`), no-op and return `{ ok: true, deduped: true }`.
- Update the watchdog scan (or add one if absent) to flip `queued`/`started` rows past `expectedBy` to `timeout` and release the queue slot.
- **Tests:** rig-based scenario tests for: enqueue writes Dispatch row, started transition, completed transition via webhook, duplicate webhook deduplicates, timeout flip releases queue slot, legacy fallback works for orphaned webhooks.

### 23.4 — Caching rollout

- Widen `AnthropicMessageParams.system` to `string | Array<{type, text, cache_control?}>` and `AnthropicToolDefinition` to allow optional `cache_control` per `references/prompt-caching.md`. Widen `AnthropicMessageResponse.usage` to include cache fields.
- **Overseer:** add `cache_control: { type: "ephemeral" }` to the last tool returned by `ToolRegistry.toAnthropicTools()`. One-line change in `lib/overseer-tools.ts`. Caches the entire system+tools prefix (~7-10K tokens, well past Sonnet 4.6's 2,048 minimum).
- **Compressed history:** widen `formatSummaryAsMessage` in `lib/chat-history-compressor.ts` to emit a content array with `cache_control` on the summary block. Extends the cached prefix through the synthetic `[Earlier conversation summary — ...]` message.
- **Per-project chat:** in `app/api/projects/[slug]/chat/route.ts`, wrap the system prompt as `[{ type: "text", text: built, cache_control: { type: "ephemeral" } }]`. Cache hits land for turn 2+ within the same project session.
- **Feature proposer:** in `lib/anthropic-feature-proposer.ts`, wrap the static system prompt with `cache_control: { type: "ephemeral", ttl: "1h" }`. Verify the prompt is over the 2,048 minimum first; if not, skip and document why.
- **Skip:** Haiku summarizer (system prompt ~100 tokens, below Haiku 4.5's 4,096 minimum). Briefing endpoint (entirely dynamic per request).
- **Snapshot tests:** new `lib/__tests__/prompt-snapshots.test.ts` containing one snapshot per cached call site: Overseer system+tools combined, per-project system, feature-proposer system. Fails loudly if anyone silently mutates a cached prefix.
- **Tests:** assert `cache_control` is set on the last tool from `toAnthropicTools()`. Mock the caller to capture the params and assert shape. Hit-rate verification via the `AnthropicUsageEvent` rows generated during integration tests.

### 23.5 — Scenario tests (the 12)

The high-yield list, all using the rig from 23.1. New file per scenario in `tests/scenarios/`:

1. `webhook-before-dispatch-event.test.ts` — webhook arrives before activity event commits → outcome still recorded (post-23.2 fix)
2. `webhook-fires-twice.test.ts` — duplicate Stop hook → one `DispatchOutcome`, one `humanTask` (deduplicated)
3. `respawn-pane-fails.test.ts` — `tmux respawn-pane` throws → queue slot released, error logged, no zombie pane
4. `dispatch-throws-next-runs.test.ts` — first dispatch throws inside `launchInPane` → next pending job runs cleanly
5. `webhook-after-project-deletion.test.ts` — project deleted, webhook fires → handler returns 200, no crash
6. `concurrent-dispatch-same-project.test.ts` — two simultaneous dispatch requests → queue serializes, both produce `Dispatch` rows
7. `hung-session-watchdog.test.ts` — dispatch with `spawnBehavior: "hang"` → `expectedBy` passes → watchdog flips to `timeout` and releases slot
8. `import-single-project-fails-mid-webhook.test.ts` — `importSingleProject` throws → webhook still 200s, no `DispatchOutcome` written
9. `concurrent-chat-session.test.ts` — extend existing test with rig — concurrent `getOrCreateChatSession` doesn't double-create
10. `osascript-quote-injection.test.ts` — custom prompt with `"`, `\`, `$` → no shell escape failure
11. `project-renamed-mid-session.test.ts` — slug mismatch on hook → webhook 200s and logs "orphaned" event
12. `dispatch-outcome-create-fails.test.ts` — Prisma create throws inside webhook → handler returns 200 with logged error

Each test ≤30 lines; all use the rig.
