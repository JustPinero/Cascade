# Request 23.5.1 — Batch Resilience + Shell Escape Verifier

## Why

Two concerns surfaced while writing 23.5's scenario tests:

1. **Partial-batch failure aborts the rest of the batch.** When a single per-project spawn throws inside `dispatchAll` / `dispatchBatch`, the lifecycle helper marks that Dispatch row `failed` and rethrows. The throw bubbles up through `queue.enqueue` (which releases the slot, drains, and rethrows) and out through the `await enqueueWithDispatchRow(...)` call inside the dispatcher's per-project loop. The loop has no try/catch, so it exits early. Already-enqueued projects keep running (their dispatch fns already completed); pending projects after the failure point never enqueue. The Dispatch row state for processed projects is correct, but the batch is silently truncated. The API caller surfaces a single 500 with the first failure's message.

2. **The shell-metacharacter scenario test is permissive.** `tests/scenarios/dispatcher-resilience.test.ts > custom prompt with shell metacharacters round-trips...` only asserts the dispatch reaches `started` and that `spawn` was called. It doesn't introspect the constructed command. That's fine as a regression-prevention smoke (broken escaping would surface as a thrown spawn), but a stricter test that asserts specific characters appear correctly escaped in the constructed command would catch silent escaping bugs that don't throw — e.g., a prompt containing `'; rm -rf /'` slipping through quote-stripping.

This slice ships fixes for both, plus tests that codify the new behavior.

## GREEN Phase (implementation)

### Files to modify — production

- `lib/claude-dispatcher.ts` — wrap each per-project `await enqueueWithDispatchRow(...)` inside `dispatchAll` and `dispatchBatch` in a try/catch. On catch, push a failure entry into `results` and continue. The lifecycle helper has already marked the Dispatch row failed by the time the throw lands, so the catch only needs to update `results` and log. Don't rethrow:

  ```ts
  // Inside dispatchBatch's per-project loop
  for (let i = 0; i < readyJobs.length; i++) {
    const { project, cmd, prompt, mode } = readyJobs[i];
    const target = paneTargets[i];
    try {
      await enqueueWithDispatchRow(prisma, {
        project,
        mode,
        prompt,
        spawnFn: async (idempotencyKey) => {
          launchInPane(target, cmd, { CASCADE_DISPATCH_ID: idempotencyKey });
          await prisma.activityEvent.create({ /* ... */ });
        },
      });
      results.push({ success: true, /* ... */ });
    } catch (err) {
      const message = err instanceof Error ? err.message : "Unknown error";
      results.push({
        success: false,
        projectName: project.name,
        projectSlug: project.slug,
        mode,
        prompt: prompt.slice(0, 300),
        ready: true,
        readyIssues: [],
        error: message,
      });
    }
  }
  ```

  Same pattern in `dispatchAll`.

- `lib/claude-dispatcher.ts:buildProjectCmd` — switch from naive `replace(/'/g, "'\\''")` style escaping to a single quote-and-escape pass that wraps the prompt as `'…'` with embedded single quotes encoded as `'\''`. Already mostly there for the projectPath; verify the prompt path in `dispatchClaude`'s temp-file flow does the same. The current code writes the prompt to a tmp file and reads via `cat`, which sidesteps direct injection via the prompt — good. The risk surface is when `customPrompt` is interpolated into commands directly elsewhere (audit and harden).

- `lib/dispatch-lifecycle.ts` — no change; lifecycle helper already isolates the throw cleanly.

### Files to modify — tests

- `tests/scenarios/dispatcher-resilience.test.ts > respawn-pane failure for one project doesn't stop the rest of the batch` — flip the assertion: was `await expect(dispatchBatch(...)).rejects.toThrow(...)`, becomes `const result = await dispatchBatch(...); expect(result.results.find(r => !r.success)?.error).toMatch(/respawn-pane/);`. The behavior changes from "throws" to "returns with per-project failure recorded."

### Files to create

- `tests/scenarios/batch-resilience.test.ts` — tests 1-3.
- `tests/scenarios/shell-escape-verifier.test.ts` — tests 5-8. Introspects `rig.spawnRecords` to verify the cmd string contains the literal characters from the input prompt, not their evaluated forms. The asserter form:

  ```ts
  it("single-quote injection cannot break out of quoting", async () => {
    const malicious = `'; rm -rf /`;
    await dispatchClaude(rig.prisma, project, "ignored", { customPrompt: malicious });
    // The prompt is written to a tmp file and read via cat. The
    // file's content is the original prompt verbatim. The cmd that
    // bash runs is `cd '/path' && CLAUDE_... claude "$(cat '/tmp/...')"`.
    // The malicious string should never escape the cat invocation.
    const cmd = rig.spawnRecords[0].args[1] ?? "";
    // Sanity: the cmd uses cat-from-tmpfile, so the malicious string
    // is not in the cmd at all (it's in the tmpfile).
    expect(cmd).not.toContain("rm -rf");
    // The cmd should reference a tmp prompt file path:
    expect(cmd).toMatch(/cat '[^']+\/cascade-prompt-[^']+'/);
  });
  ```

  The shell-escape verifier captures the **architecture-level guarantee** Cascade already has (prompts go through tmp files, never inline) rather than just escaping correctness. That's the more honest invariant.

## Out of scope

- **Refactoring the entire shell-escape strategy.** The current tmp-file approach is correct for prompts; the slice asserts that invariant rather than rewriting the escape logic.
- **Tightening dispatchTeam's failure modes.** dispatchTeam still uses the legacy queue-only path (per the 23.2 scope decision). Its single-job-failure mode is different — there's only one queue slot to lose, so the "next project doesn't run" issue doesn't apply.
- **Surfacing partial-batch results in the dashboard UI.** The API response now includes per-project status; rendering that in the UI is its own follow-up.
- **A general-purpose shell-escape utility.** If escape failures recur across call sites, factor into `lib/validators.ts` then. For now, the single quote-and-escape pattern lives inline.
