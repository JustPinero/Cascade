# Request 23.3 — Anthropic Usage Telemetry + Observability Page

## Objective

Wire token-and-cache usage logging into every Anthropic API call site that doesn't stream, then render the data as a flat table at `/observability/cache`. This must land before 23.4 (caching rollout) — without telemetry we can't prove caching is working, and we can't detect when someone destabilizes a cached prefix in a future PR.

The schema captures the four token classes that matter (`input_tokens`, `cache_read_input_tokens`, `cache_creation_input_tokens`, `output_tokens`) plus the 5-minute / 1-hour cache-creation split per `references/prompt-caching.md`. The page is intentionally minimal: sortable rows, basic filters, one computed `hitRate` column. No charts, no aggregations — those are demand-driven follow-ups when patterns emerge.

Streaming call sites (project chat, wizard) require a TransformStream wrapper to extract usage from the `message_delta` event mid-stream. That work is deferred — see "Out of scope" — because the streaming-usage pattern is cleanest to land alongside Phase 25.2.

Helper for extracting the usage fields from a response:

```ts
// In lib/anthropic-usage-log.ts — used by all call sites
export function extractUsageFields(usage: AnthropicResponseUsage | undefined) {
  return {
    inputTokens: usage?.input_tokens ?? 0,
    cacheReadInputTokens: usage?.cache_read_input_tokens ?? 0,
    cacheCreationInputTokens: usage?.cache_creation_input_tokens ?? 0,
    cacheCreation5mTokens: usage?.cache_creation?.ephemeral_5m_input_tokens ?? 0,
    cacheCreation1hTokens: usage?.cache_creation?.ephemeral_1h_input_tokens ?? 0,
    outputTokens: usage?.output_tokens ?? 0,
  };
}
```

This keeps the field-mapping logic in one place — when the API returns new usage fields in the future, only one file changes.

## Dependencies

- None hard-blocking. 23.1 (rig) is convenient for some integration-style tests but not required — the call-site usage tests can inline-mock fetch.
- 23.4 (caching rollout) **depends on this slice** — caching has no observable signal without telemetry.

## Out of scope

- **Streaming call sites** — `/api/wizard/chat` and `/api/projects/[slug]/chat` use `stream: true` against Anthropic. Logging usage from a streamed response requires a TransformStream that watches for the `message_delta` event and extracts usage at end-of-stream. Cleanest to land alongside Phase 25.2 (streaming Overseer responses), which establishes the streaming-usage pattern for the whole codebase. Mark this in `audits/debt.md` as a follow-up.
- **An Overseer tool that queries cache stats** — out of 23.3 scope. The page is the deliverable; the tool would be feature creep here. Add later if real conversational use cases emerge.
- **Charts, rollups, or aggregated dashboards.** Demand-driven. Promote when a recurring need appears.
- **Alerts on hit-rate drops.** Future. The data lands; alerting is its own slice.
- **Backfilling historical usage.** Pre-23.3 calls are not retroactively logged. The table starts empty.
