# Request 10.1 — Subagent Concurrency Queue

## Objective
Prevent memory burn on lower-RAM hosts by throttling in-flight `claude`
CLI subagents through a single dispatcher-level queue. Auto-detect a
safe default cap from host memory; allow override via env. Release slots
when the Stop webhook fires for a session.

## Acceptance Criteria → Test Mapping

| Criterion | Test |
|-----------|------|
| Central queue module exposes `enqueue(job)` and `release(jobId)` | `lib/dispatch-queue.test.ts` — unit tests for queue API |
| `detectDefaultConcurrency()` returns 1 for host memory <16GB | mock `os.totalmem()` at 8GB → expect 1 |
| `detectDefaultConcurrency()` returns 2 for 16GB ≤ memory < 48GB | mock `os.totalmem()` at 24GB → expect 2 |
| `detectDefaultConcurrency()` returns 4 for memory ≥48GB | mock `os.totalmem()` at 64GB → expect 4 |
| `CASCADE_MAX_CONCURRENT_SUBAGENTS` env overrides detected default | set env to "3" → expect 3 |
| Jobs dispatch immediately while below the cap | enqueue 2 jobs with cap=2 → both start without delay |
| Jobs beyond the cap wait in FIFO order | enqueue 3 jobs with cap=2 → job 3 waits until a release |
| `release(id)` dispatches the next queued job | release job 1 → job 3 dispatches |
| `release(id)` on unknown id is a no-op (idempotent) | release non-existent id → no throw |
| `dispatchAll` routes all jobs through the queue | spy on `queue.enqueue`, expect N calls for N projects |
| `dispatchBatch` routes all jobs through the queue | spy on `queue.enqueue`, expect N calls for N items |
| `dispatchTeam` routes the lead-agent spawn through the queue (counts as 1 slot) | spy confirms single enqueue |
| Stop webhook at `/api/webhook/session-complete/` calls `queue.release(sessionId)` | POST with sessionId → spy on `queue.release` |
| Queue is a process-wide singleton (no accidental re-instantiation) | import twice → same reference |
