# Request 13.4 — Test hardening

## What was wrong (per QA review)
- `inventory-walk.test.ts` asserted `expect(toolCallsExecuted).toBe(12)`. Adding any optional tool call to the simulated walk would have broken the test for no good reason.
- No test for the "caller throws mid-loop" case. We caught tool errors but not network/auth errors.
- No route-level test for the compressor wiring. Compressor had unit tests in isolation, but the threshold/keepRecent values + summarizer construction in route.ts were untested.

## What ships
- `inventory-walk.test.ts` — `toBe(12)` → `toBeGreaterThanOrEqual(12)`. Lower-bound stays meaningful, upper bound stops being brittle.
- New: `runToolUseLoop` propagates a caller exception out of the loop without leaking partial state (`overseer-tools.loop.test.ts`).
- New: route handler invokes the compressor when the conversation > threshold; the caller receives the compressed message list with a synthesized summary at index 0 (`route.tools.test.ts`).
- New: route handler does NOT invoke the compressor on short conversations.
- Route mock gained `chatSession.update` (caught when adding the compressor integration test).

680 → 683 (+3).
