# Phase 13 — Stability + Test Hardening

Phase 12 shipped the Overseer tool-use migration end-to-end. Phase 13
addresses everything the post-Phase-12 senior code review and QA review
surfaced. Every item below is small in code; the value is durability.

## Scope, in order

### 13.1 — Race conditions (load-bearing)
- `getOrCreateSession` had read-then-write, so two simultaneous requests
  for the same day could create duplicate sessions. Wrap in
  `prisma.$transaction`.
- `propose_dispatch` has read-modify-write of `workingMemory.proposedDispatches`.
  Add a transactional `appendToWorkingMemoryList(prisma, sessionId, key, item)`
  helper and route the tool through it.
- New concurrency tests that fire 20 parallel calls and assert: one
  session, all proposals retained.

### 13.2 — Timeout + abort discipline
- Thread `AbortSignal` through `runToolUseLoop` → `defaultAnthropicCaller`
  and `defaultSummarizer`.
- Route handler creates a 60s `AbortController` like the legacy path did,
  passes the signal through.
- Tests: abort the signal, assert fetch was cancelled.

### 13.3 — UX + cosmetics
- Truncated tool loops surface a structured summary (tools called +
  count) instead of a generic "iteration limit hit" line.
- `sseFromText` accepts the model id rather than hard-coding the
  feature-check leftover.
- Drop the no-op `body.useTools` flag (Phase 12F made it dead).
- Cache `buildDefaultRegistry()` at module level.
- `closeSession` is exported but never called — left as-is, documented
  intent ("future cron job will close inactive sessions").

### 13.4 — Test hardening
- Loosen brittle assertions in `inventory-walk.test.ts`
  (`toBe(12)` → `toBeGreaterThanOrEqual(12)`).
- New test: caller throws mid-loop — runToolUseLoop surfaces the error
  cleanly without leaking state.
- New test: route-level compressor integration (a long conversation
  that triggers compression).

### 13.5 — Make `propose_dispatch` consumable
- New `GET /api/overseer/session-state` endpoint returns the day's
  `workingMemory` + `activeFlow`. The dashboard can later read this to
  surface `proposedDispatches`. Until then the data is at least
  exposable, not write-only.
- Tests for the route + a small integration test (propose → fetch).

## What's NOT in this phase

- Live Anthropic e2e test (cost + maintenance — separate decision)
- `vitest --shuffle` flake-finder rollout (would surface flakes from
  other phases, separate scope)
- Dashboard UI changes (showing `proposedDispatches` to the user)
- Splitting concurrent ChatMessage writes (no current callers)

## Exit criteria
- All review findings either fixed, tested, or explicitly deferred.
- `validate.sh` green.
- Test count grows; flakes don't.
- Phase 12 regression test still passes verbatim.
