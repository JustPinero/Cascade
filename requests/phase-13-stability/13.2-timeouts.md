# Request 13.2 — Timeout + abort discipline

## What was wrong
Phase 12 dropped the legacy 60s `AbortController` when the
SP-injection branch was removed. The new tool path had no ceiling at
all — a hung Anthropic call (caller or summarizer) would hold the
request indefinitely.

## What ships
- `AnthropicCaller` signature now takes optional `{ signal? }`.
- `MessageSummarizer` signature now takes optional `{ signal? }`.
- `runToolUseLoop` accepts an optional `signal: AbortSignal`,
  propagates it to each `caller` invocation, and short-circuits
  with `truncated: true` if it sees `aborted` between iterations.
- `defaultAnthropicCaller` and `defaultSummarizer` thread the signal
  into their `fetch` calls.
- `compressMessagesForSession` accepts and forwards a signal.
- The route handler creates a 60s `AbortController` and threads
  the signal through both the compressor and the loop. Released in
  a `finally`.

## Acceptance
| Criterion | Test |
|---|---|
| Caller receives the abort signal | overseer-tools.loop.test.ts |
| Pre-aborted signal short-circuits the loop with truncated:true | overseer-tools.loop.test.ts |
| Test count grows by 2; nothing else breaks | full suite |

## Out of scope
- Mid-tool-execution abort (tools hold their own state; out of scope to interrupt them safely)
- Configurable timeout per request (defaults to 60s)
