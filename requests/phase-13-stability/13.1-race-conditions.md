# Request 13.1 — Race conditions

## What was wrong
Two read-modify-write patterns slipped through Phase 12:
1. `getOrCreateSession` (lib/chat-session.ts) — `findFirst` then `create` is a TOCTOU race; two concurrent requests for the same date could both miss the find and both create, producing duplicate sessions.
2. `propose_dispatch` (lib/overseer-tools-propose-dispatch.ts) — read `workingMemory.proposedDispatches`, append locally, then `mergeWorkingMemory`. Two parallel calls could both read N, both write N+1, and one proposal would vanish.

## What ships
- `getOrCreateSession` wrapped in `prisma.$transaction`. SQLite acquires a write lock for the duration; only one writer runs the create branch at a time.
- New `appendToWorkingMemoryList(prisma, sessionId, key, item)` helper in `lib/chat-session.ts`. Wraps read+write in `$transaction` and handles the corner case of an existing non-array value by initializing.
- `propose_dispatch` refactored to use the helper. The inline JSON parse + slice is gone.
- New `lib/chat-session.concurrency.test.ts`: 20 simultaneous `getOrCreateSession` calls produce ONE session, 20 simultaneous appends preserve all 20 items.

## Test count
677 → 678 (one regression test file with 5 cases; plus updates to existing route mock to handle `$transaction`).
