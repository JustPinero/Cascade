# Phase 24 — Overseer Intelligence

## Why

Phase 23 lays the regression spine; Phase 24 spends some of that runway on the two highest-leverage AI improvements that close visible gaps:

1. **Outcome-conditioned dispatch (propose-only).** The data is in the new `Dispatch` table and `DispatchOutcome` rows. The Overseer doesn't read it. The closed feedback loop — sessions → outcomes → knowledge → better dispatch decisions — is half-built today. This phase wires the second half. The Overseer surfaces *"based on the last 5 outcomes I'd recommend continue instead of audit"*; the developer still triggers. Auto-apply is a future opt-in.

2. **Tool-call observability.** The Overseer ships 14 tools with zero introspection on which gets picked, in what order, with what success. Tool-selection accuracy is the single most useful debugging signal for tool-using agents. Today it's invisible. This phase ships per-call telemetry, an Overseer tool that queries it, and a minimal `/observability/tools` page.

These are individually small features that compound. Once outcome-conditioned dispatch is real, the question becomes "is it actually helping?" — and tool-call observability lets the Overseer answer that question about itself.

## Sub-requests

### 24.1 — Outcome-conditioned dispatch (propose-only)

- New tool `query_outcome_history` in `lib/overseer-tools-outcome-history.ts`. Input: `{ slug, windowDays? }`. Output: per-mode counts, success rates, recurring signals, recent timeline (last 3-5 entries), one-line heuristic-generated `summary`. Heuristic logic in TypeScript (no model call).
- Register in `buildDefaultRegistry()` (`lib/overseer-tools-registry-default.ts`). Now 15 tools — confirm the cached tools block is still well over Sonnet 4.6's 2,048 minimum (it will be).
- Update `TOOL_PATH_SYSTEM_PROMPT` in `app/api/overseer/chat/route.ts` to add the "Outcome-conditioned proposals" section per the reference. ~40 tokens. Update the prompt snapshot test from Phase 23.4 — review the diff.
- Data source: `Dispatch` joined to `DispatchOutcome` via `dispatchId` for post-Phase-23 rows. Fall back to legacy lookup keyed by `projectSlug` for orphaned outcomes.
- **Eval fixtures (3):**
  - `repeated-audit-no-signals.json` — preconditions seed 5 prior `audit` outcomes with empty `signals`; assert tool sequence includes `query_outcome_history` then `propose_dispatch` with `mode: "continue"`; final text references the audit history
  - `continue-keeps-failing.json` — 3 prior `continue` outcomes with `outcome: blocker`; assert proposal is `investigate`
  - `first-dispatch.json` — no prior outcomes; assert `query_outcome_history` is called (returns empty), proposal proceeds with default mode, final text does NOT mention history
- **Tests:** unit tests for the heuristic summary function (8 cases — empty history, all-success, all-failure, mixed modes, etc.). Integration test that the new tool is registered and executable. Eval fixtures pass.

### 24.2 — Tool-call observability

- New `ToolCallEvent` Prisma model per `references/tool-call-observability.md` (sessionId, iteration, toolName, input truncated to 4KB, outputSize, success, errorMessage, durationMs, resultUsed default true, createdAt). Indices on `[sessionId, createdAt]`, `[toolName]`, `[createdAt]`.
- Instrument `runToolUseLoop` in `lib/overseer-tools.ts` to insert one row per `registry.execute()` call. Single Prisma insert; failure to insert never breaks the loop (best-effort).
- New tool `get_tool_call_stats` in `lib/overseer-tools-stats.ts`. Input: `{ sessionId?, windowDays?, groupBy? }`. Output: aggregated counts, success rates, latency percentiles, top tools and their frequency.
- Register in `buildDefaultRegistry()`. Updates the prompt snapshot test.
- New page `app/observability/tools/page.tsx` — server-component table per the reference. Filters: date range, toolName dropdown, success boolean, sessionId text. Columns: timestamp, sessionId, iteration, toolName, success, durationMs, outputSize. No charts. Pagination via next-page link, page size 100.
- Add a nav entry in the dashboard layout linking `/observability/cache` and `/observability/tools` (sibling pages).
- **Tests:** unit tests for the instrumentation (insert is fire-and-forget, errors don't break loop). Integration test asserting that running a 3-tool turn produces 3 rows. Eval scenario `tool-call-stats-self-query.json` — Overseer asked "which tool do you call most this week" calls `get_tool_call_stats`, returns a coherent answer.

## Out of scope

- **Auto-apply dispatch.** Propose-only must prove itself first. Auto-apply lives behind a future playbook flag (`outcomeConditionedAutoApply: true`) and routes only to mode `continue` when history is unambiguous.
- **The `resultUsed` heuristic.** Schema reserves the column; population logic deferred to a follow-up slice once basic data is flowing.
- **Charts and aggregations on `/observability/tools`.** Demand-driven. Stay flat-table until a recurring need emerges.
- **Knowledge-matcher v2 with model-assisted reranking.** Phase 25 territory if it lands; otherwise its own phase.
- **Cross-session long-term memory.** Future. Working memory remains session-scoped.
