# Request 19.2 — Engineer-channel writeback

## Why
Today, when Delamain emits `[ENGINEER]` in chat output, nothing
parses or persists it. Justin asked Del "post this to the engineer
channel" and Del's note never made it — buried in `ChatMessage`
rows. Mirrors the bug where `propose_dispatch` had no consumer
until Phase 13.5.

## Acceptance Criteria → Test Mapping

| Criterion | Test |
|---|---|
| `lib/engineer-tag-parser.ts` exports `extractEngineerMessages(text): string[]` | unit test |
| Parses `[ENGINEER]` followed by message text up to a newline boundary or end of input | unit test, multiple shapes |
| Trims surrounding whitespace from each message | unit test |
| Returns empty array when no `[ENGINEER]` tags present | unit test |
| Handles multiple `[ENGINEER]` blocks in a single response | unit test |
| Re-runs cleanly when called twice (regex lastIndex reset) | unit test |
| Route handler invokes the parser on `result.finalText` after `runToolUseLoop` returns | route test (mock caller) |
| Route handler POSTs each parsed message to `/api/engineer-channel` with `from: "delamain"` | route test (mock fetch) |
| Writeback is fire-and-forget — must not block the SSE response | route test asserts response returns even if writeback fetch hangs |
| Writeback failure logs but does not surface as a 500 | route test, mock fetch rejects |
