# Request 12A.3 — query_project tool, end-to-end

## Objective
Ship the first concrete tool — `query_project` — and wire it through
the live Overseer chat route behind an opt-in `useTools` flag. The
existing streaming-with-SP-injection path stays the default and
remains untouched. The flag-gated tool path proves the architecture
works against real route handlers and is ready for Phase B (where we
flip the default).

## Why this scope
The first tool exists primarily to validate plumbing — the next ten
tools will follow the exact same pattern, so getting this right
matters more than getting it broad. `query_project` is the right
choice for first-tool because it's the one most often asked of
Delamain ("how's medipal?"), and it has the simplest input/output
shape.

## Acceptance Criteria → Test Mapping

| Criterion | Test |
|-----------|------|
| `lib/overseer-tools-query-project.ts` exports `queryProjectTool: Tool` with name `"query_project"`, description, JSON schema requiring `slug: string` | shape assertion test |
| Tool returns `{found: true, ...}` with name, slug, status, health, currentPhase, progressScore, businessStage when project exists | unit test against in-memory prisma fixture |
| Tool returns `{found: false, slug}` when project does not exist | unit test |
| Tool surfaces parsed `progressBreakdown` from `progressDetails` JSON when present | unit test, valid JSON |
| Tool tolerates malformed `progressDetails` JSON without throwing (returns no `progressBreakdown`) | unit test, broken JSON |
| Tool surfaces `needsAttention` from `healthDetails` when present | unit test, valid JSON with needsAttention |
| Tool truncates `projectContext` to ≤200 chars and `completionCriteria` to ≤150 chars | unit test, long fixtures |
| `lib/overseer-tools-registry-default.ts` exports `buildDefaultRegistry(): ToolRegistry` registering `query_project` | unit test |
| `app/api/overseer/chat/route.ts` when `body.useTools === true`: runs the tool-use loop, returns final text via SSE | integration test with mocked Anthropic caller |
| When `body.useTools` is unset/false: the existing flow is taken (no new behavior) | integration test asserting we DO NOT call the tool registry |
| Tool-path SP is concise (≤ 1500 chars) and contains the word "tool" (forces model toward tool use) | regex assertion in the route test |
| Integration test: a single tool round-trip — model emits tool_use, tool returns project, model emits text — final SSE text contains the project name | integration test, mocked caller |

## GREEN Phase
1. **Tool definition.** `lib/overseer-tools-query-project.ts` defines
   `queryProjectTool`. Pure handler — no side effects, just a Prisma
   read. JSON-shaped output matching the criteria.
2. **Default registry.** `lib/overseer-tools-registry-default.ts`
   exports `buildDefaultRegistry()`. Future tools register here.
3. **Route integration.** Modify `app/api/overseer/chat/route.ts` to
   branch on `body.useTools === true` BEFORE the existing flow:
   - Build registry via `buildDefaultRegistry()`
   - Build a small SP (≤ 1500 chars) that instructs the model to use
     tools and not invent project state
   - Run `runToolUseLoop` with `defaultAnthropicCaller(apiKey)`
   - Wrap final text via `sseFromText` (existing helper) for the
     response body
4. **Caller injection.** The route test mocks `defaultAnthropicCaller`
   via `vi.mock` so the loop drives a canned response sequence.
