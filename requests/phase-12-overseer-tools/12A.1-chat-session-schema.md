# Request 12A.1 — ChatSession schema + backfill

## Objective
Introduce `ChatSession` as the first-class entity that owns conversation
state for the Overseer. This unblocks the migration from
"stuff-everything-in-the-system-prompt + parse-tags-from-prose" to a
tool-using agent backed by structured working memory.

This request is **schema only**. No tool framework, no behavior change,
no SP changes. Existing chat continues to work unchanged. We just have
the storage shape we need for 12A.2 onward.

## Why this first
The whole Phase 12 plan rests on `workingMemory` being a real, queryable
session-scoped JSON document. Every other layer (tool framework, flow
tracking, history compression) reads or writes this column. Schema is
the foundation; nothing downstream is safe to build until it's stable.

## Acceptance Criteria → Test Mapping

| Criterion | Test |
|-----------|------|
| Prisma `ChatSession` model exists with `id (String @id @default(cuid()))`, `startedAt`, `closedAt?`, `activeFlow?`, `workingMemory: String @default("{}")`, relation `messages` | shape assertion test in `lib/chat-session.schema.test.ts` |
| `ChatSession` has indexes on `startedAt` and `closedAt` | shape assertion |
| `ChatMessage` gains optional `sessionId: String?` with relation to `ChatSession` (optional for backwards compatibility during migration) | shape assertion |
| `ChatMessage` gains optional `toolCalls: String?` for future structured tool-call payloads | shape assertion |
| `lib/chat-session.ts` exports `getOrCreateSession(date: string): Promise<ChatSession>` — finds the latest open session whose startedAt date matches OR creates a new one | unit test with real Prisma + temp DB |
| `lib/chat-session.ts` exports `readWorkingMemory(sessionId: string): Promise<Record<string, unknown>>` — JSON-parses `workingMemory` and returns `{}` on parse failure | unit test, malformed JSON case |
| `lib/chat-session.ts` exports `mergeWorkingMemory(sessionId, patch): Promise<Record<string, unknown>>` — deep-merges `patch` into existing `workingMemory`, persists, returns new state | unit test, merge semantics |
| `lib/chat-session.ts` exports `setActiveFlow(sessionId, flow: string \| null)` | unit test |
| `lib/chat-session.ts` exports `closeSession(sessionId)` — sets `closedAt`, prevents further writes | unit test, write-after-close throws |
| `scripts/backfill-chat-sessions.ts` is idempotent: groups existing `ChatMessage` rows by `sessionDate`, creates one `ChatSession` per unique date with `startedAt = parse(sessionDate)`, `closedAt = startedAt + 1 day`, assigns `sessionId` on every existing message | unit test with seeded ChatMessage rows |
| Backfill is safe to run twice (second run is a no-op) | unit test |
| Backfill leaves `sessionDate` column intact (for one transition cycle) | unit test |
| `lib/chat-session.test.ts` integration: existing `app/api/overseer/chat/history/route.ts` read paths still return correct messages after backfill | integration test |

## RED Phase
Write `lib/chat-session.schema.test.ts` and `lib/chat-session.test.ts`
with all assertions failing. Run `pnpm test` and confirm:
- All new tests fail
- Existing tests remain at their current count (currently 428+; whatever
  baseline `pnpm test` reports before this work)

## GREEN Phase
1. **Schema change.** Add `ChatSession` to `prisma/schema.prisma` per
   the criteria. Add `sessionId` and `toolCalls` to `ChatMessage`. Run
   `pnpm exec prisma db push` (dev) and `pnpm exec prisma generate`.
2. **Helper module.** `lib/chat-session.ts` with the four exported
   functions above. Use `JSON.parse` with try/catch returning `{}` for
   `readWorkingMemory`. Deep-merge with a small recursive merge
   function (no lodash dependency).
3. **Backfill script.** `scripts/backfill-chat-sessions.ts` operates on
   the live DB by default; takes optional `--dry-run` flag. Run it
   manually after merging to populate sessionId on existing messages.
4. **Integration.** Update `app/api/overseer/chat/history/route.ts` to
   tolerate `sessionId` being present (it should already work since
   `sessionId` is optional and reads happen by `sessionDate`).
5. **Test fixture parity.** `lib/__test-utils__/prisma-push.ts` already
   pushes the canonical schema; no change needed.
