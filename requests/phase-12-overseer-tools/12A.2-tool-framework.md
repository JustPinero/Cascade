# Request 12A.2 — Tool framework + tool-use loop

## Objective
Build the registry, types, and execution loop for the Overseer's
tool-use migration. This request is the **plumbing** — no actual tools
implemented yet (that's 12A.3), no integration with the live chat
route yet (also 12A.3). Just the framework, fully tested in isolation
with an injected Anthropic caller.

## Why this scope
Splitting framework from first-tool-implementation lets us test the
loop semantics (multi-turn, tool execution, error envelopes, max
iterations) without coupling them to a specific tool's behavior. When
12A.3 lands `query_project`, it slots into the existing framework and
the wire-up is mechanical.

## Acceptance Criteria → Test Mapping

| Criterion | Test |
|-----------|------|
| `lib/overseer-tools.ts` exports `Tool<TInput, TOutput>` interface with `name`, `description`, `inputSchema`, `handler` | type-level test (compile-only) |
| `ToolContext` interface includes `prisma: PrismaClient` and optional `sessionId: string` | shape assertion test |
| `ToolRegistry.register(tool)` adds a tool; throws on duplicate name | unit test |
| `ToolRegistry.get(name)` returns the tool or undefined | unit test |
| `ToolRegistry.has(name)` returns boolean | unit test |
| `ToolRegistry.list()` returns all registered tools | unit test |
| `ToolRegistry.toAnthropicTools()` returns Anthropic's `[{name, description, input_schema}]` shape | unit test |
| `ToolRegistry.execute(name, input, ctx)` invokes the handler and wraps the output | unit test |
| `execute` returns `{ok: false, error}` on unknown tool name | unit test |
| `execute` returns `{ok: false, error}` when handler throws | unit test (handler throws, registry catches) |
| `runToolUseLoop({caller, registry, ...})` calls the injected `caller` once, returns finalText if no tool_use blocks in the response | unit test with mock caller |
| `runToolUseLoop` executes tool_use blocks and recurses up to `maxIterations` (default 8) | unit test, 2 round-trips |
| `runToolUseLoop` short-circuits at `maxIterations` and returns the conversation log + a `truncated` flag | unit test, exceeded iterations |
| Tool errors are passed back to the model as `tool_result` content with `is_error: true` (per Anthropic API spec) | unit test, mock caller asserts the user-message shape |
| The loop preserves the assistant message containing tool_use blocks (Anthropic API requires it) | unit test, message log shape |
| `maxIterations` is configurable; defaults to 8 | unit test |
| Tool input is passed through to the handler verbatim (no transformation by the loop) | unit test, handler receives raw input |
| Loop returns `toolCallsExecuted` count for observability | unit test |

## RED Phase

Confirm baseline test count remains stable plus new failures.

## GREEN Phase
1. **Types.** Define `Tool<TInput, TOutput>`, `ToolContext`,
   `AnthropicMessageParams`, `AnthropicMessageResponse`, `AnthropicCaller`,
   `ToolUseBlock`, `ToolResultBlock`, etc. Keep types minimal — just
   what the loop needs.
2. **`ToolRegistry` class.** `register`, `get`, `has`, `list`,
   `toAnthropicTools`, `execute` per criteria.
3. **`runToolUseLoop` function.** Pure async loop, parameterized by
   an `AnthropicCaller` for testability. Signature:
   ```ts
   runToolUseLoop({
     caller, model, systemPrompt, messages, registry, ctx,
     maxIterations
   }): Promise<{
     messages: AnthropicMessage[];
     finalText: string;
     toolCallsExecuted: number;
     truncated: boolean;
   }>
   ```
4. **Default Anthropic caller.** Keep this as a thin module-level
   helper that wraps `fetch("https://api.anthropic.com/v1/messages")`
   and returns the parsed JSON. Match the existing pattern in the
   codebase (no SDK dependency). NOT used by tests — tests pass
   a mock.
