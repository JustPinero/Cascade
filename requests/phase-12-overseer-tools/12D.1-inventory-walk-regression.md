# Request 12D.1 — Inventory-walk regression test

## Objective
Prove end-to-end that the architectural fix actually fixes the bug
Delamain reported via [ENGINEER]: "losing conversational context
mid-session during sprint/inventory flows… repeating questions and
losing track of answers already given."

## What the test exercises
A real Prisma test DB + a real `runToolUseLoop` driving a mock caller
that simulates a competent Delamain doing a 5-project inventory walk:

1. `set_active_flow("inventory_walk")`
2. For each project: parallel `query_project` + `update_session_memory`
3. `set_active_flow("dispatch_planning")`
4. Terminal text answer

Then a follow-up turn updates one project; the test asserts the other
four projects' confirmed state persists. That's exactly the scenario
that previously failed — confirmed values getting lost across turns.

## Acceptance
| Criterion | Test |
|---|---|
| All 5 projects' confirmed state lives in workingMemory after the walk | inventory-walk.test.ts assertion |
| activeFlow ends on "dispatch_planning" | inventory-walk.test.ts assertion |
| A subsequent update on one project preserves the other four | inventory-walk.test.ts assertion |
| The total number of tool calls executed matches the documented pattern (12) | toolCallsExecuted assertion |
| The loop reaches a terminal text turn without truncation | result.truncated assertion |

## Out of scope
- Live Anthropic call (model behavior is the model's problem; this test
  proves the architecture supports correct behavior)
- Full 25-project walk (5 covers the merge semantics; scaling to 25 is
  the same code path)
- UI surface for `proposedDispatches` from working memory (deferred —
  the structured data is there, the UI rewire is a separate concern)

## What this completes
Phase 12D's load-bearing claim: the bug is fixed. The model now has
a structured place for confirmed answers to land, AND we have a
regression test that fails if the architecture regresses.
