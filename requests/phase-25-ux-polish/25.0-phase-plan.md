# Phase 25 — UX Polish

## Why

Cascade's interaction model with the Overseer is still a buffered request-response with no visible reasoning, no streaming, and no traceable knowledge use. Three small, coherent improvements that compound into a much more trustworthy and responsive feel:

1. **Adaptive thinking on dispatch decisions.** Sonnet 4.6 supports adaptive thinking natively (manual extended thinking is deprecated on this model). Letting the model think before the most consequential call it makes — `propose_dispatch` — is a behavioral lift with effectively no client-side cost beyond preserving thinking blocks across tool turns. The eval fixtures from Phase 24 will tell us whether the proposals improved.

2. **Streaming Overseer responses.** Today the route handler buffers the full Anthropic response, then replays it as SSE. Real streaming gives time-to-first-token improvements, progressive tool-call visibility, and cleaner cancellation UX. The loop body itself doesn't change — streaming is a transport concern handled inside the caller.

3. **Inline knowledge citations.** The Overseer's harvest loop is the most distinctive thing Cascade has, and right now its value is invisible — when the Overseer answers using a `KnowledgeLesson`, the lesson disappears into paraphrase. Anthropic's Citations API gives us guaranteed, structured citations back, with `cited_text` not counting toward output tokens. Implementing this through a tool (Path B from the citations reference) keeps cache prefixes stable and lets the model decide when knowledge is relevant.

These are UX features, but each has measurable behavior consequences guarded by the eval suite.

## Sub-requests

### 25.1 — Adaptive thinking on dispatch decisions

- Sonnet 4.6 reasons internally before tool calls automatically; no `thinking` parameter required client-side.
- Widen `ContentBlock` in `lib/overseer-tools.ts` to include `ThinkingBlock`:
  ```ts
  export type ThinkingBlock = { type: "thinking"; thinking: string; signature: string };
  export type ContentBlock = TextBlock | ToolUseBlock | ToolResultBlock | ThinkingBlock;
  ```
- Update `extractTextBlocks` and `extractToolUseBlocks` filters — they already exclude non-matching types; verify they still work.
- Verify `runToolUseLoop` preserves thinking blocks across tool turns. The current line `messages.push({ role: "assistant", content: response.content })` already preserves the full content array — no logic change needed, but add a regression test confirming this.
- Update prompt snapshot tests from Phase 23.4 to allow thinking blocks in recorded responses (they're part of `response.content`, not the request).
- **Eval scenario:** `dispatch-decision-with-history.json` — preconditions: project with mixed outcome history. Assert that the model's tool sequence includes `query_outcome_history` AND that the final `propose_dispatch` arguments are coherent with the history. (This evaluates the *outcome* of thinking, not thinking presence.)
- **Tests:** unit test that `runToolUseLoop` round-trips a thinking block correctly when the assistant turn also contains tool_use. Integration test that a recorded response with thinking + tool_use is preserved unchanged in `messages[]` for the next caller invocation.

### 25.2 — Streaming Overseer responses

- New caller variant `defaultStreamingAnthropicCaller(apiKey)` in `lib/overseer-tools.ts`. Same interface as `defaultAnthropicCaller` but accepts `options.onEvent?: (e: StreamEvent) => void`. Internally:
  - Sets `stream: true` on the request body.
  - Reads `Response.body` as an SSE stream.
  - Applies events to an accumulator (per `references/anthropic-streaming-tool-use.md`) to assemble the full response.
  - Forwards each event to `onEvent` synchronously.
  - Resolves with the assembled `AnthropicMessageResponse`.
- Update `runToolUseLoop` parameters to accept an optional `onEvent` and pass it through to the caller. Defaulted to undefined → backward-compatible.
- In `app/api/overseer/chat/route.ts`, swap `defaultAnthropicCaller` for `defaultStreamingAnthropicCaller`. Plumb an `onEvent` callback that re-encodes deltas as SSE and writes to the response stream:
  - Forward `text_delta` events on text blocks directly.
  - Hide `tool_use` content_block_start/delta/stop from the client (replace with a synthetic `event: tool_call_start` for the UI's progress indicator if useful).
  - Forward `message_delta` (final usage) so client can read cache stats.
- Replace `sseFromText` with the live stream. Remove the buffer-then-replay path for the tool-use response (keep `sseFromText` for the slash-command synthetic responses where there's no live model call).
- Read `cache_creation_input_tokens` and `cache_read_input_tokens` from the final `message_delta` and write `AnthropicUsageEvent` from there (matches Phase 23.3 instrumentation but with the streamed source).
- **Tests:** unit test for the accumulator (apply events in sequence, assert assembled `content` matches). Integration test using a recorded SSE stream as input. Snapshot test on a streamed response showing all event types flowed through correctly.

### 25.3 — Inline knowledge citations

- New tool `query_knowledge_with_citations` in `lib/overseer-tools-knowledge-citations.ts`. Input: `{ query, topN? }`. Output: a Cascade-internal envelope `{ lessons: [...], document: {...} }` where:
  - `lessons` is the matched `KnowledgeLesson[]` (top-N by `knowledge-matcher`)
  - `document` is the Anthropic citations-enabled custom-content document block with `cache_control` set, ready to be injected into the next message
- Register in `buildDefaultRegistry()`. Updates prompt snapshot tests.
- The Overseer chat route, when it sees a `tool_use` for `query_knowledge_with_citations`, intercepts the result: instead of returning the raw output as a `tool_result`, it adds the document block to the **next user-role message** the loop will send back, alongside the normal `tool_result` text. This is the only call site where a tool result triggers a request-shape mutation.
- Track block-index → lesson-id mapping in the Overseer route handler for the duration of the request. When parsing the assistant's response, scan for `citations` arrays on text blocks; use the mapping to resolve each citation to a lesson ID.
- Forward the assembled response (including citation metadata) to the dashboard via the SSE stream from 25.2. Add a `citations_delta` accumulator and emit a synthetic `event: citation` SSE event for the client to render.
- Update `app/components/overseer-chat.tsx` to render citation metadata as inline `[L-42]` links to `/knowledge/lesson/L-42` (or the existing knowledge detail route). Subtle styling — small superscript, clickable.
- **Eval scenario:** `knowledge-cited-response.json` — preconditions: 3 lessons in DB. Input asks a question that should match L-2. Asserts that the model called `query_knowledge_with_citations`, that the response includes a `citation` referencing the right `start_block_index`, and that the resolved lesson ID matches L-2.
- **Tests:** unit tests for the document-block builder (correct shape, citations enabled, cache_control present, block-to-lesson mapping). Integration test that a citations-enabled response renders correctly through the SSE pipeline. UI test (or visual test) that inline links render and click through to the lesson page.

## Out of scope

- **Manual extended thinking on Sonnet 4.5.** Sonnet 4.6 adaptive is the path; manual is the fallback if adaptive doesn't deliver, and would be its own slice.
- **Files API adoption for the knowledge corpus.** Researched only in `references/anthropic-files-api.md`. Plain-text custom-content documents inline are simpler at current scale; revisit when corpus exceeds ~50K tokens regularly attached to requests.
- **Streaming on per-project chat or wizard.** Phase 25 ships streaming on the Overseer chat path only; other endpoints already stream where useful or are infrequent enough not to matter.
- **Citation editing or feedback loop.** "This citation was wrong" is a future feature.
- **Audio / voice integration with thinking.** Phase 20 voice is unchanged; thinking is invisible to the voice path.

## Definition of done

- Real streaming on the Overseer chat path; dashboard shows progressive text and tool-call indicators; cache stats land via `message_delta`.
- `query_knowledge_with_citations` registered; the Overseer uses it for knowledge-bearing answers; inline citations render in the dashboard with clickable links to lesson pages.
- Snapshot tests for prompt + tool definitions updated.
- Eval fixtures cover thinking-result coherence and citation correctness.
- handoff.md updated.
