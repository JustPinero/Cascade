# Request 41.2 — Goal-Driven Dispatch Outcomes

## Objective
Stop trusting sessions' self-reports. Every dispatch composed from a
request file gets a `/goal` completion condition derived from its
acceptance criteria; the dispatch outcome records whether the goal
evaluator (a separate model, not the working session) judged the
condition met.

## Background
`/goal <condition>` (Claude Code v2.1.139+) keeps a session working
until a Haiku-class evaluator confirms the condition against the
transcript; it works in interactive and `-p` sessions. Cascade's
dispatcher (`lib/claude-dispatcher.ts`) composes the prompt/command for
spawned sessions — the goal rides in that composition. The evaluator
reads only the transcript, so conditions must name checks whose output
the session surfaces (e.g. "scripts/validate.sh exits 0, shown by
running it").

## Acceptance Criteria → Test Mapping

| Criterion | Test |
|-----------|------|
| Dispatcher composes a /goal line from a request's acceptance criteria | unit: given a request file fixture, composed prompt contains `/goal` naming tests + validate.sh with transcript-visible checks |
| Goal condition includes a bound (e.g. "or stop after N turns") | unit: composed condition contains the turn bound |
| Dispatches without a request file (ad-hoc) skip /goal cleanly | unit: no /goal line, no crash |
| DispatchOutcome records goal state | schema: new fields (e.g. `goalCondition`, `goalAchieved`, `goalReason`); migration applies via `prisma db push` |
| Session-complete ingestion parses goal achievement from session log | unit: log fixture with an achieved-goal marker → outcome row has `goalAchieved=true` + reason; fixture without → false/null, never throws |
| Outcome recommendations (phase 40) prefer goal-verified outcomes | unit: recommendation scoring weights goal-verified success above self-reported success |
