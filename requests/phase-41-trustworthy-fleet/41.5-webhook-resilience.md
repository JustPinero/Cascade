# Request 41.5 — Session-Complete Webhook Resilience

## Objective
Stop losing Stop-hook pings. Today the fleet's Stop hooks fire-and-
forget to localhost:3000; every session that ends while Cascade's dev
server is down (or `op` signed out — a known landmine) vanishes from
dispatch outcome tracking.

## Design (two halves)
1. **Spool on failure (hook side)**: the Stop-hook script curls the
   webhook; on connection failure it appends the payload as a JSON
   line to a spool file (`~/.cascade/webhook-spool.jsonl`). The
   canonical hook script lives with Cascade and is what projects'
   settings.json invoke — update the canonical script, not 20 copies.
2. **Drain on startup + interval (server side)**: Cascade reads the
   spool on boot and on a timer, ingests entries through the same
   code path as live webhook posts, and truncates what it ingested
   (atomically — concurrent session-end writes must not be lost).

## Acceptance Criteria → Test Mapping

| Criterion | Test |
|-----------|------|
| Hook script spools when server unreachable | run script against a dead port → payload line lands in spool file |
| Hook script posts normally when server up | against a live test listener → no spool entry |
| Drain ingests spooled entries via the normal ingestion path | seed spool with 2 fixture payloads → 2 outcomes recorded, same shape as live posts |
| Drain is atomic vs concurrent writes | append during drain (simulated) → late entry survives, nothing double-ingested |
| Malformed spool lines skipped, not fatal | corrupt line among valid ones → valid ingested, corrupt quarantined + logged |
| Idempotent ingestion | same payload drained twice (idempotencyKey exists in dispatcher) → one outcome |

## RED Phase
All six rows as failing tests. The hook-script tests can execute the
script with bash against a scratch spool path + local listener.

## GREEN Phase
Goal condition: all 41.5 RED tests pass AND scripts/validate.sh exits 0.

- Projects' existing settings.json entries keep working unchanged if
  they inline the curl — the spool wrapper is for the canonical script;
  note in the report which projects still inline (fleet update is a
  follow-up request, not this one).
