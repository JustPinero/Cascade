# Request 11.2 — Feature Proposer

## Objective
Build on 11.1's catalog + ledger by generating concrete per-project
integration diffs for every detected feature gap. The Overseer chat
gains `/anthropic-feature-propose [<project-slug>...]`, which Claude
uses to draft adoption diffs grounded in each project's actual
CLAUDE.md and `.claude/settings.json` contents. Human-gated by design
— Cascade NEVER auto-applies a proposal, just renders it back in chat
for review.

## GREEN Phase
1. **Gap detection** (`findGapsForProject`) — left-join
   `UpstreamFeature` (where detector is set) against
   `ProjectFeatureUsage` for the project; return entries not in the
   ledger.
2. **Proposal generator** (`generateProposal`) — calls Claude with a
   fixed system prompt that constrains output shape: per-file
   Markdown headers + fenced code blocks, ≤50-line cap per file,
   "skip"/"need clarification" branches when the recipe doesn't fit.
   Sonnet (`claude-sonnet-4-6`) used; takes 45s timeout, raw
   `fetch` (matches the rest of Cascade's Claude integration).
3. **Per-project orchestrator** (`proposeForProject`) — loads
   project, finds gaps, reads CLAUDE.md + settings.json once, calls
   the generator per gap up to `maxFeatures`. Errors are captured
   on the result object, not thrown.
4. **Fan-out** (`proposeForAll`) — iterates every project (or
   filtered by `projectSlugs`); never lets one project's failure
   stop the rest.
5. **Markdown renderer** (`renderProposalReport`) — produces a
   summary at the top (counts + the explicit "NEVER auto-applies"
   reminder), then per-project sections, then per-feature drafts.
6. **Slash command** — extends `app/api/overseer/chat/route.ts`'s
   existing pre-Claude-call interception (added in 11.1) with a
   parallel branch for `/anthropic-feature-propose`. Same SSE
   envelope as the check command.

## Out of Scope (deferred to 11.3)
- **Persisting proposals as `FeatureProposal` rows** with
  status (proposed/accepted/rejected/applied) and resolution metadata.
  Phase 11.2 renders proposals to the chat ephemerally; you copy what
  you want into the project's Claude session and apply manually. If
  you want acceptance/rejection tracking, that's a clean 11.3 add:
  one Prisma model + a PATCH `/api/feature-proposals/[id]` route.
- **Template propagation.** When a feature reaches some adoption
  threshold across managed projects, automatically bump the kickoff
  template version to bake it in for all FUTURE projects. Real value
  but adds template-generation pressure that's better scoped
  separately.
- **Auto-apply.** Out of scope by design — every adoption stays
  human-gated.

## Notes
- The chat route's slash-command branch order is: check → propose →
  fall through to normal Claude chat. Order matters because
  `/anthropic-feature-propose` would also superficially match a
  loose "/anthropic-feature*" regex; the strict `\b` word-boundary
  in each command's matcher keeps them distinct.
- Cost control: `maxFeatures: 5` per project per call by default.
  Sonnet at ~2k tokens × 5 features × N projects can add up. Tune
  via the optional `maxFeatures` argument to `proposeForProject` /
  `proposeForAll`. Slash command always uses default.
- The proposal renderer puts the "Cascade does NOT auto-apply"
  disclaimer in the SUMMARY at the top so users see it before
  scrolling diffs. This is a load-bearing disclaimer — the moment
  diff output looks like an instruction, the user might paste it
  unchecked.
