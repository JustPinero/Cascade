# Request 11.1 — Anthropic Feature Update Check (catalog + ledger + slash command)

## Objective
Give the Overseer the ability to check whether Anthropic has shipped any
Claude / Claude Code feature that one or more of the user's projects
should adopt — and which projects already use what. This phase ships the
**catalog + ledger + on-demand check**, no diff/proposer yet (deferred
to 11.2).

Trigger: the slash command `/anthropic-feature-update-check` in Overseer
chat. Optional notification fires on `pnpm dev` startup when the local
`claude --version` differs from the value Cascade last recorded.

The catalog and tables are vendor-agnostic (`vendor` field, defaulting
to `anthropic`) so a future `11.x-openai-feature-update-check` or any
other vendor can reuse the same data model.

## GREEN Phase
1. **Prisma schema additions.** Add `UpstreamFeature`,
   `ProjectFeatureUsage`, `CascadeConfig` models per the criteria.
   Run `prisma generate` + `prisma db push`.
2. **Catalog loader / writer.** `lib/anthropic-features-md.ts` parses
   `knowledge/anthropic-features.md` into `UpstreamFeature` rows on
   first load (idempotent — only adds missing rows). Markdown shape
   defined in the seed file itself.
3. **Detection rules.** `lib/anthropic-feature-detectors.ts` — pure
   functions that take a project's parsed `.claude/` + `CLAUDE.md` +
   `package.json` and emit a list of detected feature names. One
   detector per feature (e.g. `detectsStopHook`,
   `detectsAgentTeams`, `detectsSubAgents`, `detectsMCPServers`,
   `detectsSlashCommands`). Detectors return `{featureName, signal}`
   tuples explaining what they matched.
4. **Audit driver.** `lib/anthropic-feature-check.ts`
   `auditProjectFeatureUsage(projectId)` runs every detector against
   the project, upserts `ProjectFeatureUsage` rows, deletes stale
   ones. `auditAllProjects()` iterates every `Project` row.
5. **Web fetch + Claude conversion.** `runFeatureCheck()` fetches
   each URL in `ANTHROPIC_FEATURE_SOURCES`, dedupes raw text against
   existing `UpstreamFeature.source`, runs the conversion prompt for
   each net-new chunk, parses the JSON response with a Zod schema,
   filters by `confidence ≥ 60`, surfaces the rest as candidates
   (NOT auto-written to DB).
6. **Harvester extension.** Add `[ANTHROPIC]` regex to
   `lib/knowledge-harvester.ts` next to the `[LESSON]` regex. Tagged
   entries become low-confidence `UpstreamFeature` candidates with
   `addedBy = "harvester"`.
7. **Slash command.** Extend `app/api/overseer/chat/route.ts` to
   recognize `/anthropic-feature-update-check` (case-insensitive) and
   dispatch to `runFeatureCheck`. Result is rendered into the chat
   response as a structured Markdown report (catalog deltas at top,
   usage matrix below, candidate review queue at bottom).
8. **Stop hook integration.** In
   `app/api/webhook/session-complete/route.ts`, after the existing
   escalation work, call `auditProjectFeatureUsage(project.id)`
   (best-effort; failure logged, doesn't fail webhook).
9. **Version watcher.** `lib/version-watcher.ts` exports
   `checkClaudeCodeVersion()` that runs `claude --version` (with
   timeout + try/catch), reads `CascadeConfig.lastSeenClaudeCodeVersion`,
   writes the new value if absent or unchanged, OR enqueues a system
   notification ("Claude Code updated to vX — run
   `/anthropic-feature-update-check` when ready") if changed. Hook
   into `scripts/start.sh` after `prisma db push`.
10. **Notification surface.** Reuse `ActivityEvent` with
    `eventType = "feature-check-needed"`. The Overseer chat reads
    unacknowledged events on load and surfaces them as a system
    message at the top of the conversation.
11. **Seed.** `knowledge/anthropic-features.md` ships with ~15-20
    known features. The format is defined inline in the file so it
    can grow over time without code changes.

## Out of Scope (deferred to 11.2)
- **Proposer.** Generating per-project diffs that adopt a feature.
- **Template propagation.** Bumping the kickoff template version when a
  feature is adopted across enough projects.
- **Auto-apply.** Never. Phase 11 stays advisory-only by design.

## Notes
- **Vendor-agnostic data model.** Every table has a `vendor` field
  defaulting to `"anthropic"`. A future `11.x-openai-feature-update-check`
  reuses the schema with `vendor = "openai"` and a different source
  list.
- **Confidence threshold (60) is configurable.** Surfaced as
  `FEATURE_CHECK_CONFIDENCE_THRESHOLD` env var, default 60. Tunable
  after we see real data.
- **Idempotency is non-negotiable.** Every run of `runFeatureCheck`
  with no upstream changes should produce zero DB writes (no spurious
  `discoveredAt` updates, no duplicate candidates).
- **Cascade is project-zero.** No special-casing. The audit pass
  treats Cascade like every other project.
- **Token cost.** Conversion prompts run only on net-new candidates,
  so the steady-state cost is near zero. First run on the curated
  seed file is one-time and cheap (Haiku-priced, structured output).

## Dependencies
- None. Uses existing Claude integration (`lib/claude-integration.ts`),
