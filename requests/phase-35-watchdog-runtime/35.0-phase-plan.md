# Phase 35 — Dispatch watchdog runtime [23.D5]

Closes audit-debt item `[23.D5]`. `runDispatchWatchdog` exists (`lib/dispatch-watchdog.ts`) and has unit tests, but in practice it never fires for Justin: the standalone script (`scripts/run-dispatch-watchdog.ts`) is only invoked from `bash scripts/start.sh`, and Justin runs `pnpm dev` directly. Hung dispatches hold queue slots indefinitely until process restart.

## What ships

1. **`instrumentation.ts`** at repo root — Next.js's canonical server-boot hook. Imports and calls `startDispatchWatchdog()` when the Node runtime starts.
2. **`lib/dispatch-watchdog-runtime.ts`** — exports `startDispatchWatchdog()` which:
   - No-ops when `NODE_ENV === "test"` unless `{force: true}` is passed (so test runs don't spawn intervals).
   - Single-instance guarded via a `globalThis` key (HMR re-imports + multi-worker safe).
   - Fires the watchdog once immediately, then on a 5-min interval.
   - Swallows tick errors — never throws from the timer.
3. **`predev` npm script** — runs `scripts/run-dispatch-watchdog.ts` once before `next dev` starts. Catches anything orphaned from a previous session that crashed without releasing its rows.
4. Test-only exports for the singleton state (`__stopDispatchWatchdogForTests`, `__isDispatchWatchdogRunningForTests`).

## Acceptance criteria → tests

| # | Criterion | Test |

## Out of scope

- External cron for prod deployments. Justin runs local dev only.
- Watchdog-on-write (firing when a Dispatch row goes past expectedBy). Belt-on-tick is sufficient for now.
- UI surfacing the watchdog's last run. Logged to stdout — read in dev console.
