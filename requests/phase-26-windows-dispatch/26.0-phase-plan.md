# Phase 26 — Windows Dispatch

## Why

Cascade was built on macOS. Justin moved Mac → Windows on 2026-04-15 and the dispatcher silently broke: `launchInTerminal`'s non-`darwin` branch spawns a detached `bash` with `stdio: "ignore"`, so the Claude session runs invisibly (if it runs at all). The Overseer reports success because the spawn returned and a `Dispatch` row was written, but no terminal window opens — the user sees nothing. Multi-project flows (`dispatchAll`, `dispatchBatch`, `dispatchTeam`) require `tmux`, which is not installed on Windows and a non-starter under WSL2 (the 32GB-host / ~16GB-WSL ceiling crashes under multi-Claude load — see `MEMORY.md/project_windows_environment.md`).

The fix is a first-class Windows code path that opens visible Windows Terminal (`wt.exe`) tabs running Git Bash. macOS and Linux/WSL paths stay byte-identical.

## Tradeoffs already decided

- **Shell inside wt:** Git Bash (`C:\Program Files\Git\usr\bin\bash.exe`). Preserves every existing bash-quoted dispatcher command (env injection, `$(cat tmpfile)`, `;` separators) verbatim. PowerShell would mean rewriting every cmd.
- **Multi-project layout:** one wt tab per project (no panes). Tabs are denser than separate windows and easier to read one at a time; pane layout can come later if Justin asks for it.
- **Agent teams on Windows:** out of scope. `claude --teammate-mode tmux` is hard-coupled to tmux at the Claude Code binary level. `dispatchTeam` on Windows returns an explicit failure rather than pretending.

## Sub-requests

### 26.1 — Dispatch preflight

- New module `lib/dispatch-preflight.ts`.
- Exports `checkDispatchPreflight(): Promise<PreflightResult>` where:
  ```ts
  type PreflightResult = {
    platform: "macos" | "linux" | "windows";
    ok: boolean;             // ok = every required tool found
    missing: string[];       // tool names that aren't on PATH
    tools: Record<string, string | null>; // tool → resolved path or null
  };
  ```
- Required tools per platform:
  - macOS: `claude`, `osascript`
  - Linux: `claude`, `tmux`, `bash`
  - Windows: `claude` (or `claude.cmd`), `wt.exe`, `bash` (Git Bash)
- Resolves tools using PATH lookup (`which` / `where.exe`). No side effects, no DB writes.
- Caches result in-memory for 60 s — the dispatcher hits this on every call and PATH is stable across a process lifetime.

### 26.2 — Dispatcher Windows branches

- `launchInTerminal(cmd, fullscreen, extraEnv)` gains a `platform === "windows"` branch:
  - Spawns `wt.exe` with args:
    ```
    -w 0 new-tab --title "<title>" --suppressApplicationTitle bash -c "<envPrefix><cmd>"
    ```
  - `-w 0` means "current window if any, else create one." Justin gets one Windows Terminal window with tabs accumulating instead of N separate windows.
  - `envPrefix` is the same `KEY='val' …` string the existing else-branch built — bash inside wt evaluates it the same way Linux bash does.
  - `fullscreen` is ignored on Windows (wt does not have a single-flag fullscreen equivalent; the user can F11 if they want it). Document and move on.
- `dispatchAll(prisma, mode)` and `dispatchBatch(prisma, items)` branch at the top:
  - If `detectPlatform() === "windows"`: iterate ready jobs, call `launchInTerminal` per project (via the lifecycle helper's `spawnFn`). Do **not** call `killTmuxSession`, `createPaneGrid`, `launchInPane`, or `attachTmuxSession`.
  - Otherwise: existing tmux flow unchanged.
- `dispatchTeam(prisma, items)` returns early on Windows with:
  ```ts
  { success: false, error: "Agent teams require tmux — not supported on Windows. Use single dispatch or 'Resume All' instead." }
  ```

## Out of scope

- Multi-pane wt layout (split-pane within a tab) — single-tab-per-project ships first.
- Auto-installing Windows Terminal or Git Bash. Preflight reports missing tools; install is on the user.
- WSL2 detection and routing. The whole point of this phase is to avoid WSL.
- A UI badge for platform/tool status. Out of scope — preflight is reachable from code and tests; UI surfacing is a follow-up.
