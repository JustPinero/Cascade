# Phase 28 — Preflight UI badge

## Why

Phase 26 shipped `checkDispatchPreflight()` but never wired it to the UI. If wt.exe isn't on PATH, or claude.cmd isn't installed, the user finds out by clicking Dispatch and getting a silent no-op. The preflight result needs a visible surface so missing tools are obvious before a dispatch attempt.

## Scope

- `GET /api/preflight` → returns the preflight result (`{platform, ok, missing, tools}`).
- New `<PlatformBadge />` client component in the dashboard header. Detected platform + green dot when ok / yellow dot when missing tools, hover tooltip lists the gaps.
- That's it. No settings-page deep dive yet — the badge is the discovery surface; deeper UI can come later if Justin wants it.

## Acceptance criteria → tests

| # | Criterion | Test |
|---|-----------|------|
| 1 | `GET /api/preflight` returns `{platform, ok, missing, tools}` with the right shape | `app/api/preflight/route.test.ts` |
| 2 | `<PlatformBadge />` renders "windows" / "macos" / "linux" label | `app/components/platform-badge.test.tsx` |
| 3 | Badge shows green indicator when `ok: true` | same |
| 4 | Badge shows amber indicator when `ok: false` and lists missing tools in `title` | same |
| 5 | Badge fetches `/api/preflight` on mount | same (mock fetch, assert called) |

## Out of scope

- Settings-page detailed preflight panel — follow-up if Justin asks.
- Auto-install hints (open Windows Store for wt, etc) — link to docs is enough.
- Mac/Linux UX polish beyond what the badge already does.
