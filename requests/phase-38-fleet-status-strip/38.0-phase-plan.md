# Phase 38 — Fleet status strip [P2]

Design-review recommendation P2 (`audits/design-review-2026-06-11.md` §4):
the only way to know dispatch-queue state today is to read tmux/wt panes.
With memory-gated concurrency of 1–4, knowing what's running, waiting, and
stuck is essential — and it pairs with Phase 37's self-healing work (a
"stuck" count is the human-visible alarm for slot leaks).

## What ships

1. **`GET /api/dispatch/status`** — thin read endpoint:
   ```json
   {
     "queue": { "running": 1, "pending": 0, "capacity": 2 },
     "dispatches": { "queued": 0, "started": 1, "overdue": 0 }
   }
   ```
   `queue` is the live in-process singleton (`size()` + new `capacity()`
   accessor); `dispatches` are DB row counts, where `overdue` = queued/
   started rows past `expectedBy` that the watchdog hasn't flipped yet —
   the "stuck" signal.
2. **`DispatchQueue.capacity()`** — expose the cap (separate accessor;
   `size()`'s shape is contract-tested with `toEqual`).
3. **`<FleetStatusStrip />`** — header badge next to `<PlatformBadge />`,
   polls every 15s. Shows `R/C running`, `· N queued` when waiting,
   `· K stuck` (amber) when overdue rows exist. Renders nothing until the
   first successful fetch; fetch errors keep last-known state.
