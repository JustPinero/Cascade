# Phase 40 — Outcome-driven dispatch recommendations [P3]

Design-review priority P3 ("closes the stated feedback-loop vision"). The
`DispatchOutcome` table already records every session's mode + outcome +
escalation signals. Today only the Overseer reads it (via
`query_outcome_history`); the human operator never sees the pattern. This
phase surfaces fleet-wide, actionable recommendations on the dashboard —
e.g. *"audit on medipal: 4 dispatches, 0 findings — switch to continue?"*

## Approach

1. **Pure engine** — `lib/dispatch-recommendations.ts`. A deterministic
   heuristic (no model call, mirrors `overseer-history-summary.ts`) that
   maps recent per-project outcome rows into structured `Recommendation`s.
   Reuses the same thresholds the Overseer summary already uses, so the
   human dashboard and the AI advisor agree.
2. **API route** — `GET /api/recommendations`. Queries `DispatchOutcome`
   within a window (default 14d) across the fleet, groups by project,
   feeds the engine, returns `{ recommendations: Recommendation[] }`.
   `no-store` — the widget polls.
3. **Widget** — `<DispatchRecommendations />`. Same poll-and-keep-last
   pattern as `<FleetStatusStrip />` / `<CostWidget />`. Dismissible card
   above the dashboard filters; renders nothing when there are none.

## Recommendation rules (deterministic)

- **low-signal-mode** — a diagnostic mode (`audit` / `investigate`) run
  ≥3 times in the window, every outcome `success`, no recurring signals.
  → suggest `continue`. severity `info`.
  *"audit on X: N dispatches, 0 findings — switch to continue?"*
- **failing-mode** — `continue` run ≥2 times with success rate ≤30%.
  → suggest `investigate`. severity `warn`.
  *"continue on X is failing — M% success over N — try investigate?"*
- **recurring-blocker** — any blocker-class signal (`needs-attention`,
  `human-todo`, `test-failure`) recurring in ≥50% of a mode's outcomes.
  → no suggested mode (needs the human). severity `warn`.
  *"X keeps surfacing needs-attention in continue (N dispatches)."*

Thresholds reused from `overseer-history-summary.ts` /
`overseer-tools-outcome-history.ts`: repeated-diagnostic = 3, continue
failure rate = 0.3 / min count = 2, recurring-signal = 0.5.

## Acceptance criteria

- AC1: `computeRecommendations([])` → `[]`.
- AC2: 3+ clean audits on a project → one `low-signal-mode` rec suggesting
  `continue`, message states the dispatch count and "0 findings".
- AC3: fewer than 3 audits, OR audits with a recurring signal → no
  `low-signal-mode` rec (it IS finding things / not enough data).
- AC4: `continue` ≥2 with ≤30% success → one `failing-mode` rec suggesting
  `investigate`, severity `warn`.
- AC5: a blocker-class signal recurring ≥50% of a mode → one
  `recurring-blocker` rec, severity `warn`, no suggested mode.
- AC6: recommendations are produced independently per project (multi-project
  input yields recs keyed by each slug).
- AC7: `GET /api/recommendations` returns 200 `{recommendations: []}` on an
  empty system, and aggregates real `DispatchOutcome` rows into recs.
- AC8: the route only counts outcomes within the window and parses the
  `signals` JSON defensively (malformed → treated as no signals).
- AC9: `<DispatchRecommendations />` renders nothing before first fetch and
  when the list is empty; renders each rec's message after a successful
  fetch; keeps last-known state on a later failed fetch; is dismissible.
